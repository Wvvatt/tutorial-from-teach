Lesson 0001: AML Starboard 架构导览

嵌入式媒体平台开发 · 第一课

欢迎！本课将带你完成对 AML (Amlogic) Starboard 移植代码的一次完整漫步。结束时，你将对整个代码库的布局、各模块职责、以及一条请求从 Cobalt 进入硬件的路径有清晰的认识。

1. 什么是 Starboard？

Starboard 是 Cobalt 的平台抽象层 (Porting Layer / OS Abstraction)。它的核心思想是：Cobalt（一个运行 YouTube 等应用的运行时）只通过 Starboard API 与底层平台交互，平台只需要实现这些 API 即可运行 Cobalt。

Design Principle #1
"Make APIs sufficient but minimally so" — Starboard 只提供必要的 API，任何可以在 Starboard 之上实现的功能都不应该进入 API。

Starboard 定义了约 30 个模块的 API（从 Configuration、Memory、Thread，到 Media、Player、DRM）。每个平台需要实现所有这些 API。

2. AML Starboard 移植全景

AML 的移植代码位于 third_party/starboard/aml/，遵循上游推荐的第三方移植结构。

third_party/starboard/aml/ ├── shared/ ← 所有架构共享的代码 │ ├── aml_platform.h/cc Platform Services (设备信息, 网络, 语音, HDCP, CEC) │ ├── application_aml.h/cc 应用主循环 + ESSOS 集成 │ ├── main.cc 入口点 (main 函数) │ ├── player/ 播放器实现 (15 文件) │ ├── media/ 媒体能力查询 (19 文件) │ ├── widevine/ DRM 内容保护 │ ├── audio_sink/ GStreamer 音频输出 │ ├── microphone/ 麦克风驱动 │ ├── window/ 窗口管理 │ ├── speech/ TTS 语音合成 │ ├── signal/ Freeze 信号处理 │ └── system/ 系统属性, 扩展, 路径 │ ├── arm/ ← ARM 32-bit 平台配置 │ ├── platform_configuration/ │ ├── toolchain/ │ └── configuration_public.h, atomic_public.h, ... │ └── arm64/ ← ARM 64-bit 平台配置 └── (同上结构)

3. 启动流程：从 main() 到应用循环

一切从 main.cc 开始。来看 AML 的入口：

int main(int argc, char** argv) {
  tzset();                                    // 设置时区
  gst_init_check(NULL, NULL, &error);         // 初始化 GStreamer
  InstallCrashSignalHandlers();               // 安装崩溃信号处理
  InstallDebugSignalHandlers();               // 安装调试信号
  InstallSuspendSignalHandlers();             // 安装挂起信号

  // AML Application 对象——核心主循环
  third_party::starboard::aml::shared::Application application;
  int result = application.Run(argc, argv);   // 进入事件循环
  return result;
}

关键观察：

  • GStreamer 初始化在 main() 最开头 — 媒体管道是 AML 平台的核心依赖
  • Application 继承自 QueueApplication，这是 Starboard 的标准事件队列抽象
  • application.Run() 进入事件循环，处理系统事件（ESSOS 键盘/鼠标）、Starboard 事件、定时器等

4. AML 特有的核心组件

4.1 ESSOS — 图形与输入系统

ESSOS (Ess OS) 是 Amlogic 的窗口和输入系统。AML 的 Application 类封装了 ESSOS 上下文：

class Application : public QueueApplication {
  EssCtx *ctx_;              // ESSOS 上下文
  NativeWindowType native_window_;
  std::unique_ptr<EssInput> input_handler_;

  // ESSOS 回调
  void OnKeyPressed(unsigned int key);
  void OnKeyReleased(unsigned int key);
  void OnPointerMotion(unsigned int x, unsigned int y);
  void OnDisplaySize(int width, int height);
};

ESSOS 提供键盘、指针事件回调，以及窗口创建。AML 将这些事件转换为 Starboard SbInputData 事件注入应用循环。

4.2 Amlogic::Platform — 设备服务框架

aml_platform.h 中定义了一套基于事件订阅的架构：

  • DeviceInfoService — 设备信息（分辨率、HDR支持、屏幕尺寸）
  • NetworkService — 网络状态检测（有线/无线、连接状态）
  • VoiceInputService — 语音输入（支持蓝牙和麦克风两种模式）
  • HdcpService — HDMI HDCP 状态监控（非 TV 平台）
  • CecEventHandle — HDMI CEC 远程控制（非 TV 平台）

所有服务都继承自 Amlogic::Platform::PlatformEvent::Subscriber，通过统一的事件总线接收通知。

4.2.1 双向适配器模式

每个 Service 和 Application 之间是双向通信的，这是理解 AML 平台架构的关键：

═══════════════════════════════════════ │ PlatformEvent │ │ (事件总线) │ ═══════════════════════════════════════ ↑ onEvent() ↓ subscribe() │ │ ┌────┴─────┐ ┌──────┴──────┐ │ Service │◄────►│ Application │ │ (Subscriber) │ │ └──────────┘ └─────────────┘ ↑ ↑ "反向路径" "正向路径" Service 收到事件 Application 主动查询 后回调 Application Service 的状态

正向路径 (Application → Service)： Application 需要查询设备信息时，主动调用 Service 的接口：

// Application 查询设备信息
ResolutionInfo Application::GetDisplayResolution() {
  return DeviceInfoService::Instance()->getResolution();
}
bool Application::DisplayHasHDRSupport() {
  return DeviceInfoService::Instance()->hasHDRSupport();
}
float Application::GetDisplayDiagonalSizeInInches() {
  return DeviceInfoService::Instance()->getDiagonalSizeInInches();
}

反向路径 (Service → Application)： Service 收到 PlatformEvent 后，反过来调用 Application 注入 Starboard 事件：

// DeviceInfoService: 分辨率变化 → 通知 Application
void DeviceInfoService::onDrmEvent(Event *e) {
  if (e->getType() == UEvent::EVENT_RESOLUTION_CHANGED) {
    Application::Get()->DisplayInfoChanged();
  }
}

// NetworkService: 网络状态变化 → Application 注入连接事件
void NetworkService::onEvent(Event *event) {
  if (e->getType() == NetEvent::EVENT_INTERFACE_CONNECTION_STATUS) {
    m_connectStatus = e->status;
    if (e->status)
      Application::Get()->SendNetworkConnectEvent();
    else
      Application::Get()->SendNetworkDisconnectEvent();
  }
}

// HdcpService: HDCP 失败 → Application 发送 Freeze 事件
void HdcpService::statusUpdate(Event *e) {
  Drm::getHdcpStatus(status);
  bool hdcpOk = (status.auth.hdcp1x | status.auth.hdcp2x);
  SetFreeze(!hdcpOk);  // 不保护内容时冻结应用
}

从设计模式角度看，这是 Observer + Adapter 的组合：

方向模式输入输出适配方式
正向AdapterStarboard API 调用Amlogic 平台 API同步查询适配
反向ObserverAmlogic 平台事件Starboard Application 事件异步回调适配

这也是为什么每个 Service 同时有 onEvent()（Observer 接口）和 getResolution() 之类的查询方法（Adapter 接口）—— 两套接口对应两个方向的适配需求。

4.3 GStreamer 媒体管线

AML 的媒体实现深度绑定 GStreamer：

  • player_internal.cc — 播放器使用 GStreamer pipeline
  • gst_media_utils.cc — 编解码器能力检测
  • gstreamer_audio_sink_type.cc — 音频输出基于 GStreamer
  • 依赖：gstreamer-1.0, gstreamer-app-1.0, gstreamer-video-1.0, gstreamer-audio-1.0
  • 额外库：gstsecmemallocator（安全内存分配器）、secmem

4.4 Widevine DRM

AML 实现了完整的 Widevine DRM 栈：

  • drm_system_widevine.h/cc — Widevine CDM (Content Decryption Module) 集成
  • widevine_storage.h/cc — 持久化密钥存储
  • widevine_timer.h/cc — DRM 相关的定时器
  • drm_create_system.cc — DRM 系统工厂
  • 依赖 widevine_ce_cdm_shared 库

5. 一次播放请求的完整路径

从用户点击视频到硬件解码器，经过的关键层：

Cobalt Application ↓ Starboard API (SbPlayerCreate, SbPlayerWriteSample, ...) ↓ AML Starboard Implementation ├── player/player_create.cc → 创建 GStreamer pipeline ├── player/player_write_sample.cc → 将样本送入 pipeline ├── player/player_seek.cc → Seek 操作 └── player/player_internal.cc → Pipeline 状态管理 ↓ GStreamer Pipeline (src elements, decoder, sink) ├── amlhalasink (Amlogic HW video decoder) └── gstreamer_audio_sink_type.cc (音频输出) ↓ DRM Path (if encrypted): └── widevine/drm_system_widevine.cc → 解密 → secmem → decoder ↓ Kernel Drivers (V4L2, ALSA, DRM/KMS) ↓ Amlogic SoC Hardware

6. Build System

AML 移植使用 GN 构建系统。核心在 BUILD.gn：

  • 定义 starboard_platform 静态库，汇集所有源文件
  • 通过 pkg_config 引入 GStreamer 和 WPEFramework 依赖
  • 通过参数 video_resolution 控制支持的分辨率（720p/1080p/2160p）
  • 通过参数 is_tv_platform 区分 TV 和 non-TV 平台（影响 HDCP/CEC）
  • 支持 Evergreen 模式（当 sb_is_evergreen_compatible 启用）
