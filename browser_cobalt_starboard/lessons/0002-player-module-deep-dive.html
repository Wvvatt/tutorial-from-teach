Lesson 0002: AML Player 模块深度解析

嵌入式媒体平台开发 · 第二课

第一课我们俯瞰了 AML Starboard 全景。这一课深入最核心的 Player 模块——媒体播放的核心引擎，也是 AML 移植中最大的模块（15 个文件，~2750 行）。

1. 架构概览：三层嵌套

AML Player 遵循 Starboard 的不透明句柄 (Opaque Handle) 模式，外部通过 SbPlayer 句柄操作，内部有三层：

SbPlayer (不透明句柄, typedef SbPlayerPrivate*) │ └── SbPlayerPrivate (player_internal.h) ├── player_counter (最多支持 2 个并发播放器) └── std::unique_ptr<Player> player_ │ └── PlayerImpl (内部匿名 namespace) ├── 继承 Player (虚接口: WriteSample, Seek, SetRate...) ├── 继承 DrmSystemWidevine::Observer ├── GStreamer pipeline (主数据处理) └── playback_thread (GMainLoop 事件循环)
层文件职责
SbPlayer (C API)player_create.cc 等 14 个文件Starboard 标准 API 实现，参数校验
SbPlayerPrivateplayer_internal.h桥接层，实例化 PlayerImpl，跟踪播放器数量
PlayerImplplayer_internal.cc (内部类)实际实现：GStreamer pipeline、DRM、缓冲管理
🔑 关键约束
SbPlayerPrivate::number_of_players_ 限制最多 2 个并发播放器——用于支持主播放 + PIP（画中画）。

2. 播放器创建流程

从 SbPlayerCreate 到 GStreamer pipeline 的完整链路：

--- player_create.cc:31-83 / player_internal.cc:903-1220 ---

SbPlayerCreate() →
  check callbacks, codecs, output_mode
  → SbPlayerPrivate::CreateInstance()
    if (number_of_players_ >= 2) return nullptr;
    → new PlayerImpl(this, ...)

PlayerImpl() →
  初始化 allocator_ (secmem, 如果 DRM)
  解析 max_video_capabilities → pip_ (432x240@15)
  构建 GStreamer pipeline 描述字符串
  gst_parse_launch() 创建 pipeline
  配置 appsrc (caps, callbacks, max-bytes)
  配置 video/audio sink (low-memory, pip, wait-video)
  安装 underflow 回调 + bus watch
  ChangePipelineState(GST_STATE_READY)
  创建 playback_thread → g_main_loop_run()

3. GStreamer Pipeline 构建

构造函数中动态拼接 pipeline 字符串。关键逻辑按 codec 和 PIP 状态决定：

--- player_internal.cc:1011-1096 ---

switch(video_codec):
  H264:  h264parse      (clear)
         h264secparse   (DRM)
  H265:  h265parse
  VP9/AV1: (由 secmem 处理)

switch(audio_codec):
  AAC:   aacparse ! avdec_aac
  Opus:  opusdec
  Vorbis: vorbisparse ! vorbisdec
  AC3/EAC3: ac3parse

COBALT_SET_VIDEOSINK:
  amlvideosink | westerossink (默认)

// pipeline 示例 (H264+AAC):
appsrc name=my_vsrc ! h264parse ! westerossink name=my_vsink \
appsrc name=my_asrc ! aacparse ! avdec_aac ! audioconvert ! \
  audioresample ! amlhalasink name=my_asink
🔑 PIP 模式
当 max_video_capabilities 为 "width=432; height=240; framerate=15;" 时，启用 PIP 模式：
  • 只有视频，丢弃音频 (audio_codec_ = kSbMediaAudioCodecNone)
  • sink 名称后缀为 -pip 而非 -mp
  • video sink 设置 pip=TRUE, res-usage=0
  • audio sink 设置 direct-mode=FALSE

4. 数据流：WriteSample 完整路径

Cobalt 将编码后的音视频数据送下来时，经过以下流程：

--- player_write_sample.cc:30-35 / player_internal.cc:1847-2055,1609-1765 ---

SbPlayerWriteSample2
  → player_->WriteSample(sample_type, sample_infos, count)

1. 分配 GstBuffer, 拷贝 Cobalt 数据
2. 解析 DRM info (key, iv, subsamples)
3. 检查 DRM key 是否就绪:
   ├─ 未就绪 → 存为 PendingSample, 等待密钥回调后重放
   └─ 就绪   → 继续

4. 安全路径 (DRM + video):
   ├─ gst_buffer_new_allocate(allocator_, size)  ← secmem 目标
   ├─ gst_secmem_memory_get_handle(mem)          ← 硬件句柄
   ├─ gst_buffer_add_protection_meta(buffer, ...)← 句柄挂到原始 buffer
   └─ VP9/AV1: gst_secmem_parse_vp9/av1(mem)

5. DRM 解密:
   └─ drm_system_->Decrypt(session_id, buffer, ...)
      解密引擎从 buffer (加密输入) 读取
      解密后直接写入 secmem handle 指向的内存
      (CPU 不可读, 只有硬件解码器可访问)

6. gst_app_src_push_buffer(src, buffer2)  ← 推入已含解密数据的 secmem buffer

7. 缓冲水位检查:
   ├─ > 5 秒 → 延迟 1 秒再请求
   └─ < 5 秒 → 立即 DecoderNeedsData()

6. 缓冲管理

PlayerImpl 实现了主动缓冲管理，通过定时轮询和 underflow 回调维护播放流畅性：

6.1 Underflow 处理

当 GStreamer video/audio sink 内部数据不足时触发 videoUnderFlowCallback / audioUnderFlowCallback，回调调用 ProcessUnderflow() 暂停 pipeline：

--- player_internal.cc:316-350 (callbacks), 2232-2255 (ProcessUnderflow) ---

void ProcessUnderflow() {
  if (not EOS, and not enough data) {
    ChangePipelineState(GST_STATE_PAUSED);
    pipeline_is_paused_internal_ = true;
  }
}

6.2 缓冲水位监控 (CheckVideoBufferHealth)

从 pipeline 进入 PAUSED 状态后，每 80ms 轮询一次（可被 BUFFERING_POLLING_INTERVAL 环境变量覆盖）：

关键阈值值说明
MIN_VIDEO_BUFFER_TIME250ms低于此值→暂停（数据跟不上）
MIN_AUDIO_BUFFER_TIME250ms同上，音频
ENOUGH_VIDEO_BUFFER_TIME2000ms达到此值→恢复播放
ENOUGH_AUDIO_BUFFER_TIME900ms达到此值→恢复播放

7. DRM 集成与 PendingSample 机制

加密内容的播放需要一个关键机制：密钥可能比样本到得晚。

--- player_internal.cc:2029-2041, 2532-2543, 2545-2576 ---

// 当 DRM key 未就绪时
if (session_id.empty()) {
  PendingSample ps(sample_type, buffer, iv, subsamples, ...);
  pending_samples_[key_str].push_back(ps);
  return;  // 等待密钥
}

// 当密钥就绪时
void OnKeyReady(key, key_len) {
  // 往 pipeline 发 "key-status-change" 消息
  // → HandleApplicationMessage() → WritePendingSamples()
}

// 重放挂起的样本
WritePendingSamples() {
  sort(按 serial 排序);
  for (每个 pending sample) {
    WriteSample(type, buffer, session_id, ...);  // 密钥已就绪
  }
}

密钥就绪有两个来源：

  1. DrmSystemWidevine 回调 → OnKeyReady() → pipeline 消息
  2. WriteSample 时发现密钥已就绪 → 先 WritePendingSamples 再写当前样本

8. ForceStop — 急停机制

当 Application 收到 Freeze 事件时（例如 HDMI 拔出、HDCP 失败），调用 player::ForceStop()：

--- player_internal.cc:881-898 (PlayerRegistry), 2680-2684 (ForceStop) ---

ForceStop()
  → 遍历所有 PlayerImpl, 获取 pipeline
  → 每个 pipeline 发送 "force-stop" 应用消息
     → HandleApplicationMessage():
         force_stop_ = true;
         ChangePipelineState(GST_STATE_READY);  // 立即停止
}

💡 思考题
为什么 AML 的 PlayerImpl 继承了 DrmSystemWidevine::Observer，却没有直接使用它的 OnKeyReady 来处理 pending samples，而是通过 GStreamer pipeline 应用消息间接调用？这背后有什么设计考量？
点击查看答案

核心原因：线程安全

OnKeyReady() 从 Widevine CDM 的回调线程调用。如果直接在里面调用 WritePendingSamples()，需要手动加锁保护所有 GStreamer 操作（gst_app_src_push_buffer、Decrypt 等），容易死锁或漏锁。

通过 GStreamer bus 发送应用消息：

  • 自然切换到 playback_thread（GMainLoop 所在线程）执行
  • 与 pipeline 其他消息（state-changed、EOS、error）按顺序序列化处理
  • 这是 GStreamer 的标准线程间通信模式
