Lesson 0003: Widevine DRM 内容保护

🔑 关键设计：延迟 AnnounceKeys
OnAllKeysUpdated() (第1082行) 不直接通知 Player，而是通过 SbEventSchedule(AnnounceKeys, this, 0) 在当前事件循环的下一 tick 执行。这确保了在所有密钥状态都更新完毕后再统一通知。

4. 解密路径

DrmSystemWidevine 有 两条解密路径，对应不同调用者：

4.1 Starboard 标准路径 (InputBuffer)

--- drm_system_widevine.cc:975-978 ---

SbDrmSystemPrivate::DecryptStatus DrmSystemWidevine::Decrypt(InputBuffer* buffer) {
  return kSuccess;  // 空实现，不解密
}

这是 Cobalt 上层期望的标准解密接口。在 AML 中是空实现——解密全部在 GstBuffer 路径中完成。

为什么不能在 InputBuffer 路径中解密到 secmem？因为 InputBuffer 没有元数据系统：

GstBuffer                             InputBuffer
├── data (内存)                       ├── data()  (内存指针)
├── pts/dts                           ├── size()
├── caps                              └── is_encrypted()
└── metadata                  (没有扩展机制)
      └── 可附着 secmem_handle

gst_buffer_add_protection_meta()   （无类似接口）
   ← GStreamer 设计时就支持的            ← 固定结构，不可扩展

即使 InputBuffer 能携带 secmem handle，下一层 SbPlayerWriteSample 的 SbPlayerSampleInfo::buffer 也是 const void*，同样没有传递 handle 的通道。这是 Starboard API 本身的设计限制——它假设数据流是"解密到普通内存 → 传给 player"的模型，不支持硬件安全内存路径。

4.2 GStreamer 路径 (GstBuffer)

--- drm_system_widevine.cc:809-975 ---

bool DrmSystemWidevine::Decrypt(id, GstBuffer* buffer,
    GstBuffer* sub_sample, uint32_t sub_sample_count,
    GstBuffer* iv, GstBuffer* key_id, GstCaps* caps,
    encryption_scheme, encryption_pattern)

  → DecryptImpl(id, buffer, subSample, subSampleCount,
                IV, keyID, ..., patternCrypto, patternClear,
                encryptScheme, caps)

    → DecryViaWv(id, mediaType, subSampleMapping,
                 sampleCnt, pbIV, cbIV, pbData, cbData,
                 keyId, keyIdLen, ...,
                 sec_handle)       ← secmem 句柄传递到 CDM

DecryViaWv() (第664行) 是真正的硬件解密入口，它把 secmem handle 和加密数据传给 Widevine CDM，CDM 调用 TrustZone/硬件安全引擎完成 AES 解密。

5. Key Status 管理

--- drm_system_widevine.cc:1069-1093 ---

// CDM 回调 → 收集所有就绪密钥
void OnAllKeysUpdated() {
  cached_ready_keys_.clear();
  if (event_id_ != kSbEventIdInvalid)
    SbEventCancel(event_id_);
  event_id_ = SbEventSchedule(AnnounceKeys, this, 0);
}

// 遍历所有观察者，通知每个就绪的 key
void AnnounceKeys() {
  auto ready_keys = GetReadyKeys();
  for (auto* observer : observers_) {
    for (auto& key : ready_keys) {
      observer->OnKeyReady(key.data(), key.size());
    }
  }
}

密钥状态通过 session_keys_ (unordered_map<session_id, KeysWithStatus>) 管理，每种密钥状态包括：

状态含义
kSbDrmKeyStatusUsable密钥可用，可解密
kSbDrmKeyStatusExpired密钥过期
kSbDrmKeyStatusPending密钥请求中
kSbDrmKeyStatusDownscaled分辨率降级（如 HD→SD）
kSbDrmKeyStatusError密钥错误

这里可以看到完整的 Observers 链：

Widevine CDM
  → onKeyStatusesChange()
    → DrmSystemWidevine::OnAllKeysUpdated()
      → AnnounceKeys()
        → observer->OnKeyReady()          ← PlayerImpl 就是 Observer
          → pipeline post "key-status-change"
            → HandleApplicationMessage()
              → WritePendingSamples()
                → DrmSystemWidevine::Decrypt()  ← GstBuffer 版本

6. Observer 模式的双向通信

回顾 Lesson 1 的双向适配器模式，DRM 模块也采用同样的设计：

DrmSystemWidevine (Subject) PlayerImpl (Observer) │ │ │ ← Observer::OnKeyReady() ──────────│ 密钥就绪通知 │ │ │ │ │ Decrypt(buffer, session_id, ...) ──→│ Player 调用解密 │ ← kSuccess ─────────────────────────│

与第一课的 PlatformEvent 模式不同的是，这里使用了显式的 Observer 注册 (AddObserver/RemoveObserver)，由 PlayerImpl 在构造/析构时管理生命周期。

InputBuffer 和 GstBuffer 两条解密路径有什么不同？
InputBuffer 路径GstBuffer 路径
调用者Cobalt 上层媒体管线PlayerImpl::WriteSample
调用时机SbPlayerWriteSample 之前gst_app_src_push_buffer 之前
数据载体InputBuffer (固定结构)GstBuffer (可扩展容器)
元数据无元数据系统可附着 GstProtectionMeta, secmem handle
解密输出只能写到普通内存可直接写到 secmem
后续输出SbPlayerWriteSample(const void*) — 无 handle 通道gst_app_src_push_buffer — 直接推入 pipeline

两条路径的本质差异在于 API 边界的设计粒度。Starboard 标准 API 的设计假设解密输出是普通内存，这个假设对纯软件解密场景够用，但对 secmem 这样的硬件安全内存根本不够。AML 的 GStreamer 路径绕过 API 边界，在 PlayerImpl 内部完整处理了"分配 secmem → 解密 → 推送"的闭环。

为什么不扩展 InputBuffer 让它也能携带 secmem handle？

技术上可以扩展 InputBuffer 或 SbPlayerSampleInfo 增加 metadata 字段。但问题是，即使 InputBuffer 能携带 secmem handle，谁来负责解密？

标准路径中，解密是在 Cobalt 上层调用的 Decrypt(InputBuffer*) 中完成，然后 Cobalt 再调 SbPlayerWriteSample。要做到解密到 secmem，Cobalt 上层需要知道 secmem allocator、分配 secmem buffer、传递 handle——这些都超出了 Cobalt 的职责范围。

而 AML 的路径把解密放在 PlayerImpl 中，PlayerImpl 持有 allocator_ (secmem allocator) 和 drm_system_ 的引用，天然有能力完成整条链路。这说明：硬件安全解密的控制权应该放在平台层，而不是在 Cobalt 上层。这恰恰符合 Starboard 的设计原则 #3 "Minimize the burden on the porter"——如果标准路径能处理，就无需 AML 另辟蹊径了。
