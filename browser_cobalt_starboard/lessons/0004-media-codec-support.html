Lesson 0004: Media 模块 — 编解码器能力检测

嵌入式媒体平台开发 · 第四课

前三课我们学习了 Player 和 DRM——播放和解密。这一课后退一步，看看 Media 模块（19 个文件）：Cobalt 在创建 Player 之前如何判断设备能播什么。它是整个媒体管线的看门人。

🔑 设计模式
每个文件只导出一个 Starboard API 函数（如 SbMediaIsVideoSupported）。这是 Starboard 推荐的"每函数一文件"风格，方便平台按需替换实现。

2. 编解码器支持检测

Cobalt 创建 Player 之前，会调用 SbMediaIsVideoSupported 和 SbMediaIsAudioSupported 判断传入的编解码器是否能播。AML 的实现由浅到深分三层：

2.1 GStreamer Registry 查询

--- gst_media_utils.cc:68-139 ---

GstRegistryHasElementForCodec(codec):
  1. 获取系统所有 PARSER + decoder 工厂列表
     (按 codec 类型过滤 VIDEO/AUDIO, 编译时缓存)

  2. CodecToGstCaps(codec)  →  例如 "video/x-h264, stream-format=byte-stream, alignment=nal"

  3. gst_element_factory_list_filter(decoders, caps, SINK)
     → 能找到 decoder → 返回 true

  4. 找不到 decoder → 找 parser
     parser_src_caps → gst_element_factory_list_filter(decoders, caps, SINK)
     → 能找到 parser + decoder 链 → 返回 true

  5. 都找不到 → 返回 false

关键设计：编译时缓存工厂列表 (static UniqueFeatureList parser_factories)，避免每次查询都遍历 GStreamer registry——性能优化。

2.2 Codec → GstCaps 映射

--- gst_media_utils.cc:153-229 ---

Video:                Audio:
H264  → video/x-h264  AAC   → audio/mpeg, mpegversion=4
H265  → video/x-h265          (+ channels, rate if info available)
VP9   → video/x-vp9   Opus  → audio/x-opus, channel-mapping-family=0
AV1   → video/x-av1          (+ 解析 audio_specific_config 生成 caps)
VP8   → video/x-vp8   Vorbis→ audio/x-vorbis

Theora→ video/x-theora
VC1   → video/x-vc1

2.3 视频能力判定 (SbMediaIsVideoSupported)

--- media_is_video_supported.cc:42-94 ---

SbMediaIsVideoSupported(video_codec, frame_width, frame_height, ...):

  1. decode_to_texture_required? → false (AML 不支持)

  2. COBALT_SUPPORT_AV1=n? → AV1 不支持 (环境变量覆盖)

  3. 分辨率超出屏幕? → false
     Application::Get()->GetDisplayResolution()

  4. HDR 传输特性不在白名单?
     (Bt709 / SMPTE ST 2084 / ARIB STD B67) → false

  5. HDR 视频但显示器不支持? → false
     Application::Get()->DisplayHasHDRSupport()

  6. 最终白名单:
     codec ∈ {H264, VP9, AV1} && fps ≤ 60
🔑 DeviceInfoService 的集成
第 3、5 步中，SbMediaIsVideoSupported 通过 Application::Get()->GetDisplayResolution() 和 DisplayHasHDRSupport() 调用 DeviceInfoService。这正是 Lesson 1 中"正向路径"的实例——Media API 在 Cobalt 主线程上主动查询 Service 层。

2.4 音频能力判定 (SbMediaIsAudioSupported)

--- media_is_audio_supported.cc:59-74 ---

SbMediaIsAudioSupported(audio_codec, bitrate):

  1. AC3/EAC3? → 检查 ENABLE_DOLBY 环境变量 (设备属性)
     默认支持，COBALT_FORCE_SUPPORT_xxx 风格

  2. bitrate < kSbMediaMaxAudioBitrateInBitsPerSecond
     && GstRegistryHasElementForMediaType(audio_codec)

3. DRM 与 Media 的交汇

--- media_is_supported.cc:92-100 ---

SbMediaIsSupported(video_codec, audio_codec, key_system):
  return DrmSystemWidevine::IsKeySystemSupported(key_system, video_mime)
      && DrmSystemWidevine::IsKeySystemSupported(key_system, audio_mime)

SbMediaIsSupported 是 Cobalt 在 SbPlayerCreate 之前调用的"全局"检查，确保 DRM 系统声明支持该编码的 MIME 类型。而后续的 SbMediaIsVideoSupported 和 SbMediaIsAudioSupported 是更细粒度的能力检查。

它们的关系：

Cobalt EME 选择流
  → SbMediaIsSupported(codecs, key_system)
    → DRM 是否支持这些 codec?               ← 全局
  → SbMediaIsVideoSupported(codec, w, h, ...)
    → 分辨率、HDR、白名单检查                ← 视频细粒度
  → SbMediaIsAudioSupported(codec, bitrate)
    → Dolby、GStreamer decoder、码率检查     ← 音频细粒度
  → SbPlayerCreate(...)
    → GstRegistryHasElementForMediaType()    ← 最终确认
      → PlayerImpl 构造函数

4. 缓冲区配置

剩余的 10 个 media_get_* 文件返回 Cobalt 内存管理的常量值：

函数AML 返回值含义
SbMediaGetInitialBufferCapacity0不预分配
SbMediaGetBufferAllocationUnit1MB缓冲分配粒度
SbMediaGetBufferAlignment128128 字节对齐（decoder 需求）
SbMediaGetBufferPadding1MB缓冲区尾部填充
SbMediaGetBufferStorageTypeMEMORY_POOL使用内存池管理
SbMediaGetMaxBufferCapacity64MB (or custom)最大缓冲上限
SbMediaGetVideoBufferBudget48MB (or custom)视频缓冲预算
SbMediaGetAudioBufferBudget6MB (or custom)音频缓冲预算
SbMediaGetProgressiveBufferBudget12MB (or custom)渐进式缓冲预算
SbMediaIsBufferPoolAllocateOnDemandtrue按需分配，不提前预分配

这些值直接影响 Cobalt 的内存管理和播放流畅度——缓冲区太小可能导致频繁 rebuffer，太大则浪费内存。

5. 本节在整体架构中的位置

Cobalt EME / Media Pipeline │ ├── SbMediaIsSupported() ← 检查 DRM+codec 兼容性 ├── SbMediaIsVideoSupported() ← 检查分辨率/HDR ├── SbMediaIsAudioSupported() ← 检查 Dolby/GStreamer │ │ (全部通过) ▼ SbPlayerCreate() → player_create.cc：再次 GstRegistryHasElement → PlayerImpl 构造函数 → GStreamer pipeline 创建

Media 模块是 Player 的前置检查。它确保在创建 GStreamer pipeline 之前，设备确实支持该媒体格式。重复检查（player_create.cc 中也调用 GstRegistryHasElement）是防御性设计——双重验证。

为什么 Media 模块的检查在 PlayerCreate 中还要再做一次？

SbMediaIsVideoSupported 等函数在 Cobalt 上层被调用，用于 UI 决策（如隐藏不支持的清晰度选项）。但最终裁决在 SbPlayerCreate 中——如果 GStreamer 运行时环境发生变化（比如 codec 插件被卸载），PlayerCreate 会返回 kSbPlayerInvalid。

这是防御性设计：上层检查用于用户体验，下层检查是最终的安全网。
