Lesson 0006: Audio Sink — GStreamer 音频输出

嵌入式媒体平台开发 · 第六课

第二课讨论了 Player 模块——处理编码的音视频数据。但 Cobalt 还有另一条独立的音频路径：Audio Sink。它负责将解码后的 PCM 音频数据输出到硬件，用于 UI 音效、Web Audio 等非媒体播放场景。这是与 Player pipeline 完全分离的独立 audiosink pipeline。

1. 架构概览

Audio Sink 遵循 Starboard 的 Type 模式——一个平台可以注册多种 audio sink 实现（当前只注册一种 GStreamer 实现）：

SbAudioSinkPrivate (Starboard 私有层) │ ├── Type (工厂抽象) │ └── GStreamerAudioSinkType ← 实现 Create/IsValid/Destroy │ └── SbAudioSinkPrivate instance └── GStreamerAudioSink ← 内部匿名 namespace 类 ├── GStreamer pipeline (appsrc → convert → resample → queue → sink) ├── audio_loop_thread (GMainLoop) └── frame_buffers (Cobalt 提供的 PCM 帧缓冲环)

2. 平台初始化

与 Player 不同，Audio Sink 需要显式初始化。这是在 Application 构造函数中完成的（Lesson 1 见过）：

--- gstreamer_audio_sink_type.cc:507-511, application_aml.cc:173 ---

void SbAudioSinkPrivate::PlatformInitialize() {
  auto* sink_type = GStreamerAudioSinkType::CreateInstance();
  SetPrimaryType(sink_type);          ← 注册为默认 audio sink
  EnableFallbackToStub();             ← 失败时回退到 stub 实现
}

void SbAudioSinkPrivate::PlatformTearDown() {   // application_aml.cc:178
  auto* sink_type = GetPrimaryType();
  SetPrimaryType(NULL);
  GStreamerAudioSinkType::DestroyInstance(sink_type);
}

3. GStreamer Pipeline 构建

--- gstreamer_audio_sink_type.cc:132-223 ---

GStreamerAudioSink 构造函数:

  1. 创建 audio caps: "audio/x-raw, format=S16LE|F32LE, rate=48000, channels=2"
  2. 创建 appsrc (命名 "source")
     → gst_app_src_set_max_bytes(1024 * bytes_per_frame)
  3. 创建 audio sink:
     固定 amlhalasink (direct-mode=FALSE)   ← Audio Sink 不同于 Player
  4. 构建 pipeline:
     appsrc → audioconvert → audioresample → queue → amlhalasink
  5. gst_element_set_state(GST_STATE_PLAYING)   ← 直接设为 PLAYING
  6. 创建 audio_loop_thread → g_main_loop_run()
🔑 Audio Sink 与 Player 的关键区别
  • 独立 pipeline — 不共享 Player 的 pipeline
  • 无编解码 — 只有格式转换 (audioconvert/resample)，不需要 parser/decoder
  • 直接 PLAYING — 创建后立即设为 PLAYING 状态（无 preroll 过程）
  • 无 DRM — 音频 sink 处理的是已解密的 PCM 数据

4. 数据流：从 Cobalt 帧缓冲到硬件

--- gstreamer_audio_sink_type.cc:341-448 ---

AppSrcNeedData (GStreamer appsrc need-data 回调):

  while (!enough_data_ && !destroying_):
    │
    │  update_source_status_func_(&frames_in_buffer, &offset, &is_playing, &eos)
    │  ← Cobalt 告知当前帧缓冲环中有多少帧、从哪个偏移开始
    │
    │  frames_to_write = min(1024, frames_in_buffer, ring_buffer_remaining)
    │
    │  if (is_playing && frames_to_write > 0):
    │    │
    │    │  1. gst_buffer_new_allocate(nullptr, frames_to_write * bytes_per_frame)
    │    │  2. gst_buffer_fill(buffer, 0, frame_buffers[0] + offset, size)
    │    │     ← 从 Cobalt 帧缓冲环拷贝 PCM 数据到 GstBuffer
    │    │  3. GST_BUFFER_TIMESTAMP = total_frames * SECOND / sample_rate
    │    │  4. gst_app_src_push_buffer(appsrc, buffer)
    │    │  5. consume_frame_func_(frames_to_write, ...)  ← 通知 Cobalt 已消费
    │    │
    │  if (!is_playing || frames_in_buffer <= 0):
    │    SbThreadSleep(5ms)  ← 没有数据时等待

数据来源是 Cobalt 的 frame_buffers——一片环形缓冲区，由 Cobalt 上层往里面填 PCM 数据，audiosink 从中消费。

5. 线程模型

Audio Sink 有自己的 audio_loop_thread + 独立的 GMainLoop：

--- gstreamer_audio_sink_type.cc:267-276 ---

AudioThreadEntryPoint:
  g_main_context_push_thread_default(main_loop_context_)
  g_main_loop_run(mainloop_)

// appsrc need-data 回调在这个线程中执行
// bus message 回调也在这个线程中执行
// 整个 GStreamer audiosink pipeline 只在这个线程上运行
🔑 总共多少线程？
回顾第二课：Player 有自己的 playback_thread + GMainLoop。所以：
  • Cobalt 主线程
  • Player playback_thread (1 个或 PIP 时 2 个)
  • Audio Sink audio_loop_thread (1 个)
总共 3-4 个线程，每个有独立的 GMainLoop。

6. 销毁序列

销毁过程需要等待 appsrc_exit_ 标志——这是典型的优雅退出：

--- gstreamer_audio_sink_type.cc:225-264 ---

~GStreamerAudioSink():
  1. 设置 destroying_ = true
  2. gst_app_src_set_max_bytes(appsrc, 1)  ← 唤醒可能阻塞等待的 appsrc
  3. while (!appsrc_exit_) SbThreadSleep(5ms)
     等待 appsrc need-data 回调退出（appsrc_exit_ 在回调中设为 true）
  4. 超时 1 秒后退出 mainloop
  5. SbThreadJoin(audio_loop_thread_)
  6. 清理 GStreamer 资源

7. DUMP_PCM_TO_FILE — 内嵌调试功能

--- gstreamer_audio_sink_type.cc:420-440 ---

#if defined(DUMP_PCM_TO_FILE)
  SbFile file = SbFileOpen("/tmp/sound{timestamp}.pcm",
                           kSbFileOpenAlways | kSbFileWrite, ...);
  SbFileSeek(file, kSbFileFromEnd, 0);
  SbFileWrite(file, frame_buffers[0] + offset, frames_to_write * bytes);
#endif

编译时开启后，所有推到 audiosink 的 PCM 数据都会被写入 /tmp/sound*.pcm 文件。这是嵌入式开发中常用的原始音频调试手段——可以在 PC 上用 ffplay -f s16le -ar 48000 -ac 2 /tmp/sound*.pcm 回放。

8. AutoAudioSinkChildAddedCallback

--- gstreamer_audio_sink_type.cc:460-477 ---

当 amlhalasink 内部创建子 element 时:
  if (子 element 是 GST_AUDIO_BASE_SINK):
    g_object_set(sink, "buffer-time", 50ms)
    g_object_set(queue, "min-threshold-time", 50ms * 1000ns)

这是为低延迟音频优化的——50ms 的 buffer-time 和 min-threshold-time，平衡了延迟和抗卡顿。
