Lesson 0007: Hang Detector + libcobalt — 系统守护与平台控制

嵌入式媒体平台开发 · 第七课

这一课介绍两个系统级组件：Hang Detector——嵌入式必备的看门狗机制，防止线程死锁/卡死；libcobalt——RDK/WPEFramework 控制 Cobalt 生命周期的外部 C API。

PART A: Hang Detector — 看门狗

1. 概念

嵌入式设备上，线程卡死（因死锁、死循环、GStreamer 挂起等）不能导致设备无响应。HangMonitor 是每个线程的健康标记，HangDetector 线程定期巡检。

2. 注册与心跳

--- hang_detector.cc:184-215 ---

HangMonitor::HangMonitor("Application"):
  expiration_time_ = now + check_interval   ← 初始过期时间
  GetHangDetector()->AddMonitor(this)       ← 注册到巡检列表

void HangMonitor::Reset():
  expiration_time_ = now + check_interval   ← 重置过期时间 (心跳)
  expiration_count_ = 0                     ← 清零连续过期计数

// 复位间隔 = check_interval / 2
// 每隔这个时间点主线程必须调用 Reset()
// 这是"两倍冗余"——连续两次漏掉才判定为过期

在代码中的具体使用：

对象负责的线程Reset 频率
Application::hang_monitor_Cobalt 主线程 (ess_timer_fd poll)每 16ms
PlayerImpl::hang_monitor_playback_thread (GLib timer source)同 get_interval/2

3. 巡检与杀进程

--- hang_detector.cc:109-148 ---

HangDetector::DoWork() 的循环:

  for (;;) {
    WaitTimed(check_interval)  ← 等待巡检间隔 (默认 30s)

    for (每个 monitor):
      if (expiration_time > now): continue  ← 还活着, 跳过

      if (++expiration_count < 6):
        print_action()          ← 打印警告, 继续等
      else:
        kill_action():          ← 连续 6 次过期 → 杀进程
          kill(pid, SIGFPE)     ← 先发浮点异常信号
          sleep(1 min)          ← 等 1 分钟
          kill(pid, SIGKILL)    ← 还没结束 → 强制杀
          sleep(1 min)
          give up               ← 彻底放弃
  }
🔑 分阶段杀进程的策略
连续 6 次过期 = 默认 30s × 6 = 3 分钟 后认定卡死。先用 SIGFPE（可被捕获，触发 crash dump），再等 1 分钟给 crash handler 时间保存诊断信息，最后用 SIGKILL 强制结束。这是嵌入式系统常见的优雅崩溃策略。

PART B: libcobalt — 外部控制 API

5. RDK 如何控制 Cobalt

libcobalt.cc 定义了一组 extern "C" 函数，是 RDK/WPEFramework 调用 Cobalt 的唯一入口。它们通过 APIContext 单例等待 Application 就绪后执行：

--- libcobalt.cc:40-132 ---

struct APIContext:
  condition_ + mutex_ + running_ 标志

  WaitForApp(lock):
    while (running_ == false)
      condition_.Wait()        ← 阻塞直到 Application 就绪

  OnInitialize():
    running_ = (Application::Get() != nullptr)
    condition_.Broadcast()     ← 唤醒所有等待的调用

6. API 一览

--- libcobalt.cc:160-211 ---

extern "C":

  SbRdkSuspend():
    → SendBlurEvent → SendConcealEvent → SendFreezeEvent
    ← 硬暂停: 隐藏 UI, 停止播放

  SbRdkResume():
    → SendUnfreezeEvent → SendRevealEvent → SendFocusEvent
    ← 恢复

  SbRdkPause():
    → Application::Blur(semaphore) → Wait
    ← 软暂停: 只是去焦点, 不停止播放

  SbRdkUnpause():
    → Application::Focus(semaphore) → Wait
    ← 恢复焦点

  SbRdkQuit():      → Application::Stop(0)
  SbRdkIsResumed(): → IsResumed()
  SbRdkSendLink():  → SendLink(deepLink)

  SbRdkRegisterNotify(callback):
    → 注册 notify callback (用于 RDK 回调通知 suspend 等)

  cobalt_oem_notify_suspend(reason, param):
    → 调用注册的 callback
🔑 Suspend vs Pause 的区别
  • Suspend: Blur + Conceal + Freeze → 完全隐藏 + 停止 Player (ForceStop)
  • Pause: Blur → 仅失去焦点, 播放继续
这是 TV 场景中"换台" vs"按 Home"的区别——换台应该暂停, 按 Home 应该冻结。

7. 回调注册

--- libcobalt.cc:197-208 ---

static libCobaltCallback cobalt_plugin_callback = nullptr;

SbRdkRegisterNotify(callback):    ← RDK 注册回调
  cobalt_plugin_callback = callback

cobalt_oem_notify_suspend(reason, param):  ← 系统通知 Cobalt
  cobalt_plugin_callback(reason, param)    ← 转发到 RDK

这是一个回调反转模式——RDK 控制 Cobalt（Suspend/Pause/Resume），但 Cobalt 也需要通知 RDK 某些事件（如 OEM suspend），双方通过这个回调通信。

8. 初期化时序

--- application_aml.cc:173 ---

Application::Initialize() 被调用时:
  libcobalt_api::Initialize()         ← APIContext::OnInitialize()
    running_ = true                   ← 解锁 WaitForApp()
    condition_.Broadcast()            ← 唤醒所有在 WaitForApp 中阻塞的调用

这意味着: 在 Application::Initialize() 完成之前,
所有 SbRdkSuspend/Pause/Quit 调用都会阻塞。
不会出现 Application 尚未初始化就被外部控制的竞态。
