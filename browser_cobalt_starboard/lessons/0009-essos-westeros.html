Lesson 0009: ESSOS + Westeros — 显示与图形架构

嵌入式媒体平台开发 · 第九课

前面八课我们一直在看 Cobalt 内部怎么跑。这一课往前走一层，看看 Cobalt 下面是什么——Amlogic 的显示和输入栈是怎么和 Cobalt 交互的。

PART A: ESSOS — Amlogic 显示/输入抽象层

1. 什么是 ESSOS

ESSOS (ESS OS) 是 Amlogic 专有的图形和输入抽象库 (libessos)。它为应用提供统一的接口，屏蔽底层是直接操作 DRM/KMS、还是经过 Wayland 合成器、还是其他显示系统。

--- 不同平台下的 ESSOS 底层适配 ---

TV 平台:    Cobalt → libessos → DRM/KMS (直接控制显卡)
RDK 平台:    Cobalt → libessos → Westeros Wayland → DRM/KMS
纯 Linux:   Cobalt → libessos → X11

ESSOS 的设计理念类似于 Starboard 的设计理念——"对上层提供统一 API，对下层适配不同实现"。

2. ESSOS 核心 API

3. ESSOS 在 AML 代码中的使用

回顾第一课的 Application 构造函数：

--- application_aml.cc:101-140 ---

Application::Application() {
  ctx_ = EssContextCreate();               ← 创建 ESSOS 上下文
  EssContextInit(ctx_);                     ← 初始化

  // 注册 4 个回调监听器
  EssContextSetTerminateListener(ctx_, this, &terminateListener);
  EssContextSetKeyListener(ctx_, this, &keyListener);
  EssContextSetPointerListener(ctx_, this, &pointerListener);
  EssContextSetSettingsListener(ctx_, this, &settingsListener);
}

事件循环中 ESSOS 的驱动（application_aml.cc:196）：

--- application_aml.cc:192-196 ---

Application::PollNextSystemEvent():
  if ((now - ess_loop_last_ts_) > 16ms):     ← 每 16ms 一次 (60fps)
    ess_loop_last_ts_ = now;
    EssContextRunEventLoopOnce(ctx_);          ← 驱动 ESSOS 一次, 触发键盘/指针回调
🔑 16ms 的来历
kEssRunLoopPeriod = 16666 microseconds ≈ 16.7ms ≈ 60fps。这个值定义了 Cobalt 的渲染帧率。

4. ESSOS 的 EGL 集成

ESSOS 还提供了 OpenGL 集成：

--- system_egl.cc (RDK contrib):142-166 ---

SbEglGetDisplay():
  EssCtx *ctx = Application::Get()->GetEssCtx();

  // 查询 ESSOS 的 EGL 显示类型
  EssContextGetEGLDisplayType(ctx, &display_type);

    → 使用 eglGetPlatformDisplayEXT(EGL_PLATFORM_WAYLAND_EXT, ...)
    → 否则回退到标准 eglGetDisplay(display_id)

ESSOS 知道底层是 Wayland 还是 DRM，然后帮 Cobalt 选择正确的 EGL 初始化路径。

PART B: Westeros — RDK 的 Wayland 合成器

5. 什么是 Westeros

Westeros 是 RDK 的轻量 Wayland 合成器。从它的 GitHub README：

Westeros is a light-weight Wayland compositor library. It supports normal, nested, and embedded compositors.

三种运行模式（由启动参数决定）：

Normal   模式: Cobalt → Westeros → 屏幕           (直接显示到硬件)
Nested   模式: Cobalt → Westeros → 另一个合成器     (作为 Wayland 客户端)
Embedded 模式: 外部应用 → Westeros → 嵌入到自己的 UI (嵌入集成)

在 RDK 机顶盒/TV 场景中，Westeros 运行在 Normal 模式，全屏控制显示器。

6. Westeros 在 Cobalt 中的位置

Westeros 在 Cobalt 架构中有两个切入点：

--- 两条通往 Westeros 的路径 ---

路径 A: Cobalt 渲染 (ESSOS)
  Cobalt Application
    → EssContextCreateNativeWindow() → libessos
      → Westeros Wayland protocol → westeros compositor
        → DRM/KMS → GPU

路径 B: 视频渲染 (GStreamer)
  Player::WriteSample → gst_app_src_push_buffer
    → GStreamer pipeline → westerossink (GStreamer element)
      → Westeros Wayland buffer → westeros compositor
        → DRM/KMS → GPU (硬件 overlay)

视频走 westerossink，UI 走 ESSOS → Wayland。两条路径在 Westeros 合成器汇合，由硬件合成到同一帧。

但二者的 buffer 类型完全不同：

UI (ESSOS → Wayland)视频 (westerossink)
内存类型REE (普通内存)Secure DMA-BUF (安全内存)
分配器libessos 内部 shm/dma 分配gst_secmem_allocator (secmem)
CPU 可读?是否 (只有硬件解码器可访问)
传输方式Wayland shm buffer / dma-bufDMA-BUF (WESTEROS_SINK_AMLOGIC_USE_DMABUF=1)
受 HDCP 保护?否是 (通过 TrustZone)

这和第三课的 secmem 架构一致——视频解密后的数据在安全内存中，通过 DMA-BUF 直接传递到 Westeros 合成器，全程不经过 CPU 可读的 REE 内存。WESTEROS_SINK_AMLOGIC_USE_DMABUF=1 就是这个零拷贝安全路径的开关。

7. Westeros 环境变量

从 starboard/rdk/launcher.py:195-209 中提取的 Westeros 启动配置：

--- launcher.py:195-209 ---

WESTEROS_GL_USE_BEST_MODE=1           ← 使用显示器最佳模式
WESTEROS_GL_MAX_MODE=3840x2160        ← 最大分辨率 4K
WESTEROS_GL_GRAPHICS_MAX_SIZE=1920x1080 ← UI 图形分辨率限制
WESTEROS_GL_USE_AMLOGIC_AVSYNC=1      ← Amlogic 硬件音视频同步
WESTEROS_SINK_AMLOGIC_USE_DMABUF=1    ← DMA-BUF 零拷贝视频帧
WESTEROS_SINK_USE_FREERUN=1           ← 自由运行模式 (不同步)
WESTEROS_SINK_USE_ESSRMGR=1           ← 使用 ESS 资源管理器
WESTEROS_GL_USE_REFRESH_LOCK=1        ← 刷新锁定
WESTEROS_GL_USE_UEVENT_HOTPLUG=1      ← HDMI 热插拔支持
ESSOS_NO_EVENT_LOOP_THROTTLE=1        ← 禁用 ESSOS 事件循环节流
LD_PRELOAD=/usr/lib/libwesteros_gl.so  ← OpenGL 拦截器
变量作用
WESTEROS_GL_USE_AMLOGIC_AVSYNC=1启用 Amlogic 硬件 AV 同步——Westeros 将视频帧的 PTS 传给硬件，由硬件控制 vsync 时机
WESTEROS_SINK_AMLOGIC_USE_DMABUF=1视频帧通过 DMA-BUF 传递——零拷贝，解码器输出直接映射到 Westeros 的 compositor buffer
WESTEROS_GL_GRAPHICS_MAX_SIZE=1920x1080UI 渲染上限——4K 显示器上 UI 只渲染 1080p 然后硬件放大，节省 GPU
LD_PRELOAD=libwesteros_gl.soOpenGL 拦截器——接管所有 EGL/OpenGL 调用，routing 到 Westeros 的 Wayland EGL
🔑 关键设计：UI 和视频的不同分辨率路径
Westeros 将 UI (Cobalt 渲染) 和视频 (GStreamer 解码) 分开处理：
  • UI: 渲染到 1920x1080 → 硬件放大到 3840x2160（节省 GPU）
  • 视频: 解码器直接输出 4K → DMA-BUF 零拷贝到合成器 → 硬件 overlay
这是嵌入式设备的标准做法——UI 不需要 4K 精度，但视频必须原生 4K 以保持画质。
