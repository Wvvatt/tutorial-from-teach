Lesson 1

浏览器视频 API

<video> / MSE / EME 是浏览器视频系统暴露给网页的三层入口。先把这三层的职责分清，后面的 Blink、demuxer、decoder 和 compositor 才能串起来。

本课目标：看到一个视频播放请求如何从 HTML API 变成资源加载、缓冲、媒体事件和后续 MSE/EME 分工。实际播放使用本地 MP4，避免远端 DASH/CDM 环境影响课堂演示。

4. Progressive 模型

最简单的 <video> 播放走 progressive download。浏览器持有 URL，网络栈可用 byte-range 请求跳转到接近目标时间的文件区域。

<video controls src="movie.mp4"></video>

GET /movie.mp4
Range: bytes=1048576-2097151

206 Partial Content
Content-Range: bytes 1048576-2097151/52428800

这个模型简单，但它不能让网页自己做自适应码率、直播窗口、广告拼接和精确的 DRM 策略控制。

5. MSE 模型

MSE 把资源选择权交给 JavaScript。浏览器不再直接理解 DASH/HLS 清单，而是只要求被 append 的 segment 符合容器和 codec 约束。

const mediaSource = new MediaSource();
video.src = URL.createObjectURL(mediaSource);

mediaSource.addEventListener("sourceopen", async () => {
  const sb = mediaSource.addSourceBuffer(
    'video/mp4; codecs="avc1.42E01E, mp4a.40.2"'
  );
  const init = await fetch("init.mp4").then(r => r.arrayBuffer());
  sb.appendBuffer(init);
});

6. EME 模型

EME 不负责下载媒体数据，也不规定具体 DRM。它只标准化网页、媒体元素和 CDM 的会话协议。

const access = await navigator.requestMediaKeySystemAccess(
  "com.widevine.alpha",
  [{
    initDataTypes: ["cenc"],
    videoCapabilities: [{ contentType: 'video/mp4; codecs="avc1.42E01E"' }]
  }]
);
const mediaKeys = await access.createMediaKeys();
await video.setMediaKeys(mediaKeys);
