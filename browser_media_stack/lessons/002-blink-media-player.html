Blink 层桥接

一个 <video> 标签不会直接进入 decoder。它先在 Blink 中变成 HTMLMediaElement 的状态机，再通过 WebMediaPlayer 接口进入 Chromium media pipeline。

2. HTMLMediaElement 的职责

职责说明
属性和状态维护 readyState、networkState、duration、seeking。
资源选择处理 src、<source>、preload 和 load algorithm。
事件派发向 JS 发出 loadedmetadata、canplay、waiting、timeupdate 等。
平台桥接通过 WebMediaPlayer 调用底层 media 实现。

3. readyState 状态机

HAVE_NOTHING
  load() / src changed
HAVE_METADATA
  duration, tracks, dimensions known
HAVE_CURRENT_DATA
  current frame available
HAVE_FUTURE_DATA
  at least next frame available
HAVE_ENOUGH_DATA
  enough buffered data for playback
关键点：JS 看到的是 HTMLMediaElement 状态，但底层的 readyState 变化通常来自 demuxer、decoder、buffering 和 compositor 的异步回调。

4. WebMediaPlayerImpl 的两条 load 路径

Progressive

  • URL 由浏览器网络栈读取
  • DataSource 提供 byte range
  • 通常进入 FFmpegDemuxer 或平台 demuxer

MSE

  • JS 下载和选择 segment
  • SourceBuffer 推送字节
  • 进入 ChunkDemuxer 和 SourceBufferStream
HTMLMediaElement::LoadResource()
  -> CreateWebMediaPlayer()
  -> WebMediaPlayerImpl::Load()
     -> choose demuxer
     -> media::Pipeline::Start()
        -> Demuxer::Initialize()
        -> Decoder::Initialize()
        -> Renderer::Initialize()

5. Mojo/IPC 边界

Blink 运行在 Renderer 进程，但现代 Chromium 的硬件解码、CDM 和 GPU 合成通常跨进程。WebMediaPlayerImpl 持有的对象很多只是代理。

边界典型用途
Renderer -> GPUMojoVideoDecoder、VideoFrameCompositor、shared image
Renderer -> CDMContentDecryptionModule、license session
Renderer -> Browser全屏、媒体会话、权限和页面生命周期
