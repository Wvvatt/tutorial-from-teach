Decoder 抽象层

从 VDA 到 media::VideoDecoder 再到 MojoVideoDecoder，Chromium 把“解码”从 GPU 纹理专用接口演进成跨平台、跨进程、可接 CDM 的统一抽象。

1. 能力检测演示

这个检测器使用浏览器原生 API，不加载远端视频。它用于观察当前浏览器对 codec、MediaCapabilities 和 Key System 的暴露情况。

2. VDA：旧的硬件解码抽象

VideoDecoderAccelerator 面向早期 ChromeOS 硬件解码，输出以 GPU Picture/texture 为核心。这让硬解接入直接，但也把接口紧耦合到 GPU 管线。

class VideoDecoderAccelerator {
  bool Initialize(const Config& config, Client* client);
  void Decode(const BitstreamBuffer& bitstream_buffer);
  void AssignPictureBuffers(...);
  void ReusePictureBuffer(int32_t picture_buffer_id);
  void Flush();
  void Reset();
};
局限：VDA 假设输出是 GPU texture，不适合软件解码、Android MediaCodec 异步模型、CDM 解密组合和更清晰的流控。

3. media::VideoDecoder：统一接口

现代接口输出 VideoFrame，不强制 GPU texture。软件解码、VA-API、V4L2、D3D11、VideoToolbox 和 MediaCodec 都能实现同一套契约。

class VideoDecoder {
  void Initialize(const VideoDecoderConfig& config,
                  bool low_delay,
                  CdmContext* cdm_context,
                  InitCB init_cb,
                  const OutputCB& output_cb,
                  const WaitingCB& waiting_cb);

  void Decode(scoped_refptr<DecoderBuffer> buffer,
              DecodeCB decode_cb);

  void Reset(ResetCB reset_cb);
};

4. MojoVideoDecoder：跨进程代理

Renderer 进程中的 MojoVideoDecoder 通常只是客户端代理，实际硬件解码器在 GPU 进程或平台服务中。

Renderer Process
  WebMediaPlayerImpl
    -> MojoVideoDecoder
       -- Mojo IPC -->
GPU Process
  MojoVideoDecoderService
    -> VaapiVideoDecoder / V4L2VideoDecoder / D3D11VideoDecoder
       -> VideoFrame(Mailbox / GpuMemoryBuffer)

5. 加密内容：DecryptingVideoDecoder

EME 内容需要 CDM 参与。L3 常见路径是 CDM 解密 clear bitstream，再交给普通 decoder；L1 则尽量把 key、解密后数据和 secure frame 留在硬件安全路径里。

[Encrypted DecoderBuffer]
  -> DecryptingVideoDecoder
     -> CdmContext / CDM decrypt
     -> downstream VideoDecoder
     -> VideoFrame
     -> Compositor
