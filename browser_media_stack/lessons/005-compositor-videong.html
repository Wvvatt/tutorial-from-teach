Compositor + VideoNG

解码后的 VideoFrame 还不是屏幕像素。Chromium 要在每一帧决定：这块视频直接提升为硬件 overlay，还是作为纹理进入 GPU 合成。

1. 两条显示路径

路径数据流优势代价
OverlayVideoFrame -> hardware plane -> scanout低功耗、低延迟、少带宽受 plane 数量、格式、变换和遮挡限制
GPU 合成VideoFrame -> texture -> compositor render pass支持任意 UI、圆角、滤镜、变换更多 GPU 和内存带宽

3. VideoFrameCompositor

VideoFrameCompositor 连接 media pipeline 和 cc compositor。它接收 VideoRenderer 输出的帧，并实现 cc::VideoFrameProvider 让 compositor 按 vsync 拉取当前帧。

VideoRendererImpl
  -> VideoFrameCompositor::PaintFrame(frame)
  -> cc::VideoLayerImpl::AppendQuads()
  -> viz::CompositorRenderPass
  -> OverlayProcessor::Process()
  -> overlay or GPU compositing

4. OverlayProcessor 的决策条件

  • 候选层必须是轴对齐矩形，不能有复杂 3D transform。
  • 候选层没有被不可合成的 UI 遮挡，或遮挡可以由硬件 plane 处理。
  • 硬件支持该 buffer format、modifier、缩放比例和 z-order。
  • 可用 overlay plane 数量足够。
  • 受保护内容还要求 secure overlay 和输出保护。

5. Hint frame

VideoNG 中的 hint frame 是轻量元数据：真正的视频帧到来前，先告诉 compositor 即将出现的视频尺寸、格式和 overlay 候选信息。这样可以提前做 promotion 决策，减少第一帧抖动。

6. 常见退化场景

UI 覆盖

弹幕、字幕、浮层按钮可能导致 overlay demotion。

变换复杂

旋转、透视、滤镜和圆角通常需要 GPU 合成。

plane 不足

多个视频或 PiP 会竞争有限 overlay planes。
