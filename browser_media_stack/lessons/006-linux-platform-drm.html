Lesson 6

Linux 平台层

在嵌入式 Linux 上，浏览器视频性能最终取决于 dma-buf、Wayland、KMS plane、V4L2/GStreamer 与 compositor 的协作。

1. 三种“层”

层在哪里含义
cc::LayerChromium Renderer/Compositor网页内容的合成层，包括 video layer。
Wayland surfaceWayland client/compositor客户端提交给 compositor 的窗口或子表面。
KMS planeLinux kernel/display controller显示控制器硬件扫描输出通道。

2. dma-buf 贯穿链路

dma-buf 是一个跨设备共享缓冲区的文件描述符。视频解码器、GPU 和 display controller 可以围绕同一块 buffer 协作，避免 CPU 拷贝。

V4L2 decoder
  -> decode into dma-buf
  -> EGL imports dma-buf as image/texture
  -> Wayland linux-dmabuf creates wl_buffer
  -> KMS atomic commit attaches framebuffer to plane
  -> display controller scanout

3. Overlay plane 预算模拟

很多低成本 SoC 只有 1-2 个可用 video overlay。这个小模拟用于快速判断多路视频会如何回退到 GPU 合成。

Plane budget

4. KMS atomic commit

Atomic commit 允许一次提交多个 plane 的 framebuffer、位置、尺寸和 z-order，避免中间状态造成撕裂。

drmModeAtomicSetProperty(req, plane_id, "FB_ID", fb_id);
drmModeAtomicSetProperty(req, plane_id, "CRTC_ID", crtc_id);
drmModeAtomicSetProperty(req, plane_id, "SRC_X", src_x << 16);
drmModeAtomicSetProperty(req, plane_id, "SRC_W", src_w << 16);
drmModeAtomicSetProperty(req, plane_id, "CRTC_W", dst_w);

drmModeAtomicCommit(fd, req, DRM_MODE_ATOMIC_NONBLOCK, NULL);

5. Wayland linux-dmabuf

浏览器作为 Wayland client 把 dma-buf fd 交给 compositor。compositor 再决定导入 GPU 合成，或尝试把它直接放到 KMS overlay plane。

zwp_linux_buffer_params_v1_add(params, fd, plane, offset, stride, modifier_hi, modifier_lo);
wl_buffer *buffer = zwp_linux_buffer_params_v1_create_immed(
  params, width, height, DRM_FORMAT_NV12, flags);
wl_surface_attach(surface, buffer, 0, 0);
wl_surface_commit(surface);

6. GStreamer 对接视角

WebKit/WPE 和一些 Chromium 嵌入式分支会用 GStreamer 连接网络、demux、decode 和 sink。关键是让硬件解码器输出 memory:DMABuf，再导入 EGL/Wayland。

souphttpsrc ! qtdemux ! h264parse ! v4l2h264dec !
  video/x-raw(memory:DMABuf),format=NV12 !
  glimagesink
嵌入式优化目标：让视频帧从 VPU 到 display controller 尽量保持 dma-buf handle 传递，而不是回到 CPU 内存再上传。
