# ESP32 + OV2640 课程路线

这条路线从已经能运行的串口心跳程序出发，每课只增加一个主要概念。每个阶段都有一个真实设备上的可观察成果和一个小改动，完成后再进入下一课。

## 阶段一：建立设备直觉

1. **编译、烧录、串口反馈**
   - 实践：修改心跳周期，编译上传，确认串口和 GPIO2 变化。
   - 通过标准：能解释 `pio run`、`upload`、`monitor` 的边界。
2. **串口输入与时间**
   - 实践：用串口命令改变 LED 闪烁速度，同时保持心跳输出。
   - 通过标准：能区分阻塞式 `delay` 和基于 `millis()` 的定时。
3. **内存与故障反馈**
   - 实践：打印堆内存，制造并观察一次受控失败。
   - 通过标准：能从日志判断程序是否重启、卡住或内存不足。

## 阶段二：把设备放进网络

4. **Wi‑Fi 连接状态机**
   - 实践：连接当前局域网，打印 IP、重试次数和断线状态。
5. **HTTP 状态接口**
   - 实践：浏览器访问 `/health`，读取 uptime、RSSI 和 free heap。
6. **服务器与设备的协议边界**
   - 实践：用一个最小 Python server 接收 JSON，比较 GET、POST 和状态码。

## 阶段三：让 OV2640 工作

7. **识别板型和摄像头引脚**
   - 实践：确认开发板型号、OV2640 排线方向和 D0-D7、XCLK、PCLK 等引脚。
8. **抓取第一张 JPEG**
   - 实践：初始化 `esp_camera`，抓取一帧，串口打印尺寸和字节数。
9. **单张照片 HTTP 接口**
   - 实践：ESP32 提供 `/capture`，浏览器直接显示 JPEG。
10. **实时预览的代价**
   - 实践：比较单张抓拍和 MJPEG stream，观察帧率、内存和网络负载。

## 阶段四：最终项目

11. **间隔拍照上传**
   - 实践：每隔 N 秒向同网段 server `POST /images` 上传 JPEG。
12. **服务端保存与播放**
   - 实践：按时间保存图片，网页按时间顺序播放最近照片。
13. **可靠性与可诊断性**
   - 实践：断线重连、上传超时、失败重试、有限缓存和结构化日志。
