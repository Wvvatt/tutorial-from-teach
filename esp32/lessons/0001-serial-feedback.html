让设备反馈成为调试工具

第 1 课 · 预计 20 分钟 · 真实设备实践

本课目标：你将把一次“程序运行了”变成三条证据：串口心跳、GPIO2 状态和可重复的编译/烧录命令。这是之后判断 Wi‑Fi 或摄像头问题的基线。

1. 先观察，不改代码

在终端执行：

cd /home/teng.wang/Code/Teach/esp32/esp32-test
pio run
pio device monitor

如果监视器只显示横幅，按一下板上的 EN/RESET。你应该看到 heartbeat 每秒递增，gpio2 在 HIGH/LOW 之间交替。

2. 做一个最小改动

打开 src/main.cpp，把：

constexpr uint32_t kIntervalMs = 1000;

改成：

constexpr uint32_t kIntervalMs = 2500;

然后执行：

pio run -t upload

重新打开监视器，验证两次心跳约相隔 2.5 秒。不要凭“烧录命令成功”判断程序更新；用输出时间间隔证明它真的运行了新版本。

3. 回忆检查

不要查资料，先写下：为什么这次只改应用代码就能工作，而课程后面的摄像头实验通常需要同时关心分区、PSRAM 和 GPIO 引脚？

4. 通过标准

你完成了本课，当你能把“编译失败、烧录失败、程序启动但逻辑不对”三种情况分别指出证据来源，并把心跳间隔改成 2.5 秒后在串口中验证。

下一步

下一课会加入输入和定时器：用串口命令改变闪烁周期，同时保持串口心跳。这会让你理解为什么摄像头采集和网络上传不能用一串长时间阻塞的 delay。
