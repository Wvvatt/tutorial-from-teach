读懂内存和故障反馈

第 3 课 · 预计 25 分钟 · 真实设备实践

本课成果：你能在串口里看到当前可用堆内存，知道这串数字代表什么，并能通过一次受控失败判断程序是“没起来、卡住了、还是在重启”。这会直接服务后面的 Wi-Fi 和摄像头课。

1. 先回答记忆题

Free heap: 351468 bytes 里的 heap 是什么？先用一句话说出它代表的资源，再说出为什么我们要关心它。

2. 打印一条内存基线

打开 esp32-test/src/main.cpp，在 setup() 里、Serial.begin() 之后加上这一行：

Serial.printf("Free heap: %lu bytes\n",
              static_cast<unsigned long>(ESP.getFreeHeap()));

如果你已经有心跳输出，这条信息应该在启动后很快出现。它给你一个“程序刚启动时可用 RAM 还剩多少”的起点。

3. 做一次受控失败

在 setup() 末尾加一段只做一次的探测，不要放进 loop()：

constexpr size_t kProbeBytes = 1024 * 1024;
void* probe = malloc(kProbeBytes);
if (probe == nullptr) {
  Serial.printf("malloc(%lu) failed as expected\n",
                static_cast<unsigned long>(kProbeBytes));
} else {
  Serial.printf("malloc(%lu) unexpectedly succeeded\n",
                static_cast<unsigned long>(kProbeBytes));
  free(probe);
}
为什么这样做：你不是在破坏程序，而是在确认“内存不足”会以什么形态出现在日志里。后面抓摄像头帧或开网络时，这类证据比猜测更重要。

4. 学会区分三种输出

现象常见含义下一步看什么
只有监视器横幅，没有程序输出程序没真正跑起来，或串口参数不对EN/RESET、baud、rts/dtr
启动后正常打印，但随后停住卡在某个阻塞调用或死循环里最近改过的函数、delay、等待条件
反复出现启动信息，然后又重来一次崩溃后在重启，可能是异常或内存问题复位原因、最后一条日志、是否有栈回溯

5. 编译、烧录、验证

cd /home/teng.wang/Code/Teach/esp32/esp32-test
pio run -t upload
pio device monitor

启动后，你应该至少看到两类证据：一条 Free heap 记录，和一条 malloc 的成功或失败记录。把这两条和当前心跳输出放在一起看，不要只看单独一行。

6. 小挑战

把内存打印再放一次到 loop() 中，但只在程序启动后前 3 次心跳里打印。目的不是刷屏，而是观察“启动时”和“运行一会儿后”数值有没有变化。先想清楚需要哪一个计数状态，再动手改。

7. 回忆检查

如果串口里反复出现启动信息，你更应该先怀疑“程序在重启”，还是“串口读错了”？请说出你的判断依据，至少包含一条日志证据。

通过标准

你能解释 Free heap 的含义，知道 ESP.getFreeHeap() 返回的是字节数，并能从日志里区分“正常运行”“阻塞”“重启”和“内存不足”的大致形态。

遇到异常时，把启动日志、最后一条输出和你改过的代码片段一起发来。内存问题靠证据判断，不靠感觉。
