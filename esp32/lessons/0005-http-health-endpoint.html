让浏览器读取 ESP32

第 5 课 · 预计 25 分钟 · 一个请求闭环

本课成果：在 Mac 上访问 ESP32 的 /health，得到 uptime、RSSI 和 free heap；访问错误路径时得到 HTTP 404。这个接口将成为摄像头接入前的网络诊断入口。

0. 开课门槛

先确认串口出现 wifi: CONNECTED ip=...。如果仍显示 403 Forbidden 作为 SSID 或持续 retry，先修正 src/secret.h，否则 Mac 不知道该访问哪个 IP。

1. 先预测

第 4 课中，为什么 wifiLoop() 必须快速返回？把同一个原则迁移过来：如果 server.handleClient() 一直得不到控制权，浏览器的请求会发生什么？先在心里作答，再继续。

2. 建立 HTTP 服务

在 include 区加入：

#include <WebServer.h>

在匿名命名空间中创建监听 80 端口的服务：

WebServer server(80);
bool httpStarted = false;

3. 定义 /health 的响应

在 wifiBegin() 前加入：

static void handleHealth() {
    String body;
    body.reserve(96);
    body += "uptime_ms=" + String(millis()) + "\n";
    body += "rssi=" + String(WiFi.RSSI()) + "\n";
    body += "free_heap=" + String(ESP.getFreeHeap()) + "\n";
    server.send(200, "text/plain; charset=utf-8", body);
}

这里第一次看见 HTTP 响应的三部分：状态码 200、内容类型、响应体。API 签名可在本机 WebServer.h 的 send() 中核对。

4. 只在拿到 IP 后启动

加入一个短函数：

static void httpLoop() {
    if (WiFi.status() != WL_CONNECTED)
        return;

    if (!httpStarted) {
        server.on("/health", HTTP_GET, handleHealth);
        server.onNotFound([] {
            server.send(404, "text/plain; charset=utf-8", "not found\n");
        });
        server.begin();
        httpStarted = true;
        Serial.println("http: listening on port 80");
    }

    server.handleClient();
}

在主 loop() 中，放在 wifiLoop() 后面：

wifiLoop();
httpLoop();
heartbeatLoop();

5. 编译、烧录、测量

cd /Users/watt/Code/Teach/esp32/esp32-test
pio run -t upload
pio device monitor

记下串口打印的 IP，例如 192.168.1.42。另开终端：

curl -i http://192.168.1.42/health
curl -i http://192.168.1.42/no-such-path

第一次应为 HTTP/1.1 200 OK，第二次应为 HTTP/1.1 404 Not Found。连续请求两次 /health，确认 uptime_ms 增大、心跳仍继续。

6. 回忆检查

server.on()、server.send()、server.handleClient() 分别负责什么？请在答案中使用“路由”“响应”“控制权”。

通过标准

  • /health 返回 200 和三项实时状态。
  • 未知路径返回 404。
  • 连续请求时心跳不中断。
  • 能解释为什么 handleClient() 必须反复获得控制权。
