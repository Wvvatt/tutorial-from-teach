用 HTTP 返回一张 JPEG

第 9 课 · 预计 25 分钟 · 浏览器可见结果

本课成果：ESP32 提供 GET /capture。浏览器每刷新一次，摄像头抓取一张新 JPEG，并以 image/jpeg 响应；连续刷新三次，照片正常且心跳继续。

1. 把两条已知链路接起来

server.handleClient()
    ↓ 找到 GET /capture
handleCapture()
    ↓
fb_get() → HTTP headers → JPEG body → fb_return()

server.on() 负责注册路由，handleClient() 负责让请求获得处理机会。新知识只有一个：HTTP 正文不是文字，而是 fb->buf 指向的 JPEG 原始字节。

2. 响应的三个组成部分

部分本课内容作用
状态码200抓拍成功
Content-Typeimage/jpeg告诉浏览器按 JPEG 解码
Content-Lengthfb->len告诉浏览器正文有多少字节

3. 增加抓拍路由处理函数

把下面函数放在 httpLoop() 之前：

static void handleCapture() {
    if (!cameraReady) {
        server.send(503, "text/plain; charset=utf-8", "camera not ready\n");
        return;
    }

    camera_fb_t* fb = esp_camera_fb_get();
    if (fb == nullptr) {
        server.send(500, "text/plain; charset=utf-8", "capture failed\n");
        return;
    }

    server.sendHeader("Cache-Control", "no-store");
    server.setContentLength(fb->len);
    server.send(200, "image/jpeg", "");
    server.sendContent(reinterpret_cast<const char*>(fb->buf), fb->len);

    Serial.printf("http: GET /capture 200 bytes=%zu\n", fb->len);
    esp_camera_fb_return(fb);
}
所有权边界：sendContent() 使用 fb->buf 作为响应正文，所以此前不能归还缓冲；发送调用完成后立刻 fb_return()，让缓冲池能接收下一帧。不要调用 free(fb)。

4. 注册路由

在 httpLoop() 的 server.begin() 之前加入一行：

server.on("/health", HTTP_GET, handleHealth);
server.on("/capture", HTTP_GET, handleCapture);
server.onNotFound([] { server.send(404, "text/plain; charset=utf-8", "not found\n"); });

注册仍然只执行一次。真正处理请求仍靠 loop() 不断调用 server.handleClient()。

5. 编译、烧录、查看照片

pio run -t upload
pio device monitor

从串口找到 ESP32 的 IP，例如 192.168.1.195，在浏览器访问：

http://192.168.1.195/capture

也可以在 Mac 保存响应并查看响应头：

curl -D - http://192.168.1.195/capture -o capture.jpg
file capture.jpg

关键证据应包括：

HTTP/1.1 200 OK
Content-Type: image/jpeg
Content-Length: ...

http: GET /capture 200 bytes=...

6. 三次刷新实验

  1. 打开 /capture，确认浏览器显示照片。
  2. 移动摄像头或改变画面，然后刷新两次。
  3. 确认画面变化、每次字节数可能不同，并且串口心跳仍继续。

Cache-Control: no-store 防止浏览器直接复用旧照片，让每次刷新更可靠地到达 ESP32。

7. 故障签名

  • 503 camera not ready：摄像头初始化失败，先看启动时的十六进制错误码。
  • 500 capture failed：初始化成功但取帧失败，检查供电、排线及缓冲是否归还。
  • 浏览器下载乱码文本或不显示：检查 Content-Type 是否为 image/jpeg。
  • 照片截断：检查 Content-Length 是否等于 fb->len，以及是否过早归还缓冲。
  • 第一次成功、后续卡住：优先检查所有成功路径是否执行 fb_return()。

通过标准

  • 浏览器能直接显示 /capture 返回的照片。
  • 响应头包含 image/jpeg 和正确长度。
  • 连续刷新三次都成功，照片内容发生变化。
  • 心跳继续，并能解释发送和归还之间的所有权边界。
