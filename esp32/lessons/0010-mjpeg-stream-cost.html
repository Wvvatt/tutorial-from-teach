实时预览的代价

第 10 课 · 预计 30 分钟 · 对照实验

本课成果：增加最小 GET /stream MJPEG 预览，实测 FPS、每秒发送字节和剩余堆；同时证明同步流处理会占住 Arduino loop()，让心跳和其他 HTTP 路由暂时失去控制权。

1. MJPEG 不是一种新的视频编码

MJPEG 流就是在一个长期不结束的 HTTP 响应中，不断放入完整 JPEG。每帧前都有边界和自己的 multipart 分段头：

Content-Type: multipart/x-mixed-replace; boundary=frame

--frame
Content-Type: image/jpeg
Content-Length: 12345

[一张 JPEG]
--frame
Content-Type: image/jpeg
Content-Length: 11980

[下一张 JPEG] ...

浏览器读到一个边界就解码一帧，因此不需要 H.264 那种帧间压缩和视频解码器。

2. 增加最小流处理函数

把下面函数放在 httpLoop() 之前：

static void handleStream() {
    if (!cameraReady) {
        server.send(503, "text/plain; charset=utf-8", "camera not ready\n");
        return;
    }

    WiFiClient client = server.client();
    client.print(
        "HTTP/1.1 200 OK\r\n"
        "Content-Type: multipart/x-mixed-replace; boundary=frame\r\n"
        "Cache-Control: no-store\r\n"
        "Connection: close\r\n\r\n");

    uint32_t reportStartedMs = millis();
    uint32_t frames = 0;
    uint32_t bytesSent = 0;

    while (client.connected()) {
        camera_fb_t* fb = esp_camera_fb_get();
        if (fb == nullptr) {
            Serial.println("stream: capture failed");
            break;
        }

        const size_t frameLength = fb->len;
        client.printf(
            "--frame\r\nContent-Type: image/jpeg\r\n"
            "Content-Length: %zu\r\n\r\n",
            frameLength);
        const size_t written = client.write(fb->buf, frameLength);
        client.print("\r\n");
        esp_camera_fb_return(fb);

        if (written != frameLength) {
            Serial.printf("stream: short write %zu/%zu\n", written, frameLength);
            break;
        }

        ++frames;
        bytesSent += static_cast<uint32_t>(written);
        const uint32_t now = millis();
        const uint32_t elapsedMs = now - reportStartedMs;
        if (elapsedMs >= 1000) {
            Serial.printf("stream: fps=%lu bytes_per_sec=%lu free_heap=%u\n",
                          static_cast<unsigned long>(frames * 1000 / elapsedMs),
                          static_cast<unsigned long>(bytesSent * 1000 / elapsedMs),
                          ESP.getFreeHeap());
            reportStartedMs = now;
            frames = 0;
            bytesSent = 0;
        }

        delay(1);
    }

    client.stop();
    Serial.println("stream: client disconnected");
}
先归还，再判断短写：只要 fb_get() 成功，这条路径最终就必须归还帧缓冲。即使网络写入失败，也不能把缓冲留在池外。

3. 注册路由

在 server.begin() 之前加入：

server.on("/stream", HTTP_GET, handleStream);

烧录后访问：

http://你的ESP32地址/stream

如果浏览器不能直接显示 multipart 响应，用一个最小 HTML 页面测试：

<img src="http://你的ESP32地址/stream">

5. Wi‑Fi 为什么能运行，心跳却停止

Arduino loopTask（当前配置固定在 CPU 核 1）
  └─ httpLoop()
      └─ server.handleClient()
          └─ handleStream()
              └─ while (client.connected())  ← 一直停在这里

Arduino 的 delay(1) 最终调用 FreeRTOS 的 vTaskDelay()，暂时阻塞当前 loopTask；恢复后仍从同一个调用栈继续。它不是协程切换，也不会替你再次调用 heartbeatLoop()。

在当前双核配置中，Wi‑Fi 驱动任务固定在 CPU 核 0，本来就能与核 1 的 loopTask 并行；如果多个任务位于同一核心，vTaskDelay() 也会给其他就绪任务调度机会。心跳不同：heartbeatLoop() 不是独立任务，只是 Arduino loop() 顺序调用的普通函数。handleStream() 不返回，它就永远拿不到控制权。

6. 单张与实时流的取舍

/capture/stream
连接时间一张后结束长期保持
网络流量用户请求时产生持续产生
帧缓冲周转一次借还连续借还
当前同步实现很快回到 loop占住 loop
适合任务定时拍照、上传人工实时预览

最终任务是“间隔拍照上传”，因此不需要把 MJPEG 当作最终架构。本课的价值是看见实时预览的成本。

通过标准

  • 浏览器能看到 MJPEG 实时画面。
  • 记录至少 5 秒的 FPS、bytes/s 和 free heap。
  • 流打开时 /health 超时，关闭后恢复。
  • 能解释 CPU 调度与 Arduino loop() 控制权的区别。
