间隔拍照上传

第 11 课 · 预计 40 分钟 · 最终项目第一条纵向链路

本课成果：按串口 u 开启上传后，ESP32 每 10 秒抓取一张 JPEG，向 Mac 的 POST /images 发送原始字节；Mac 验证类型、长度、JPEG 首尾标记并返回 201 Created。本课只验证，不保存图片。
开课前关闭 /stream 标签页：第 10 课的同步流循环会占住 Arduino loop()；它不返回，下面的 uploadLoop() 就不会运行。

1. 先明确协议

项目约定
方法与路径POST /images
请求正文一张 JPEG 的原始字节，不是 JSON，也不是 Base64
Content-Typeimage/jpeg
成功状态码201 Created
序号X-Image-Sequence 请求头

直接传 JPEG 比 Base64 更小：Base64 通常会把体积增加约三分之一，而且双方还要额外编码、解码。

2. 让 Mac 接收并验证，但暂不落盘

把 esp32-test/server/server.py 替换为：

import json
from http.server import BaseHTTPRequestHandler, ThreadingHTTPServer


MAX_IMAGE_BYTES = 500_000


class Handler(BaseHTTPRequestHandler):
    def send_json(self, status, payload):
        body = (json.dumps(payload) + "\n").encode("utf-8")
        self.send_response(status)
        self.send_header("Content-Type", "application/json")
        self.send_header("Content-Length", str(len(body)))
        self.end_headers()
        self.wfile.write(body)

    def do_POST(self):
        if self.path == "/events":
            length = int(self.headers.get("Content-Length", "0"))
            body = self.rfile.read(length)
            print("event:", body.decode("utf-8"))
            self.send_json(201, {"accepted": True})
            return

        if self.path != "/images":
            self.send_error(404)
            return

        content_type = self.headers.get_content_type()
        length = int(self.headers.get("Content-Length", "0"))
        sequence = self.headers.get("X-Image-Sequence", "?")

        if content_type != "image/jpeg":
            self.send_json(415, {"error": "expected image/jpeg"})
            return
        if length < 4 or length > MAX_IMAGE_BYTES:
            self.send_json(413, {"error": "invalid image size"})
            return

        image = self.rfile.read(length)
        complete = len(image) == length
        jpeg = image.startswith(b"\xff\xd8") and image.endswith(b"\xff\xd9")
        if not complete or not jpeg:
            self.send_json(422, {"error": "invalid jpeg"})
            return

        print(f"image: seq={sequence} bytes={len(image)} jpeg=valid")
        self.send_json(201, {
            "accepted": True,
            "sequence": sequence,
            "bytes": len(image),
        })


server = ThreadingHTTPServer(("0.0.0.0", 8000), Handler)
try:
    print("receiver: listening on 0.0.0.0:8000")
    server.serve_forever()
except KeyboardInterrupt:
    print("\nreceiver: stopped")
finally:
    server.server_close()

3. Mac 先自测协议

启动接收器：

cd esp32-test
python3 server/server.py

另开终端，从 ESP32 的单张接口拿到真实 JPEG，再上传给本机：

curl http://192.168.1.195/capture -o /tmp/esp32-test.jpg
curl -i -X POST http://127.0.0.1:8000/images \
  -H 'Content-Type: image/jpeg' \
  -H 'X-Image-Sequence: mac-test' \
  --data-binary @/tmp/esp32-test.jpg

只有看到 201 Created 和 jpeg=valid 才继续。此时已证明服务端协议正确，后面出错就优先检查 ESP32 客户端。

4. 在 ESP32 中加入上传状态

在全局状态区加入，把 IP 换成 Mac 的局域网地址：

constexpr char kUploadUrl[] = "http://192.168.1.140:8000/images";
constexpr uint32_t kUploadIntervalMs = 10000;
uint32_t lastUploadAttemptMs = 0;
uint32_t uploadSequence = 0;
bool uploadEnabled = false;

如果不确定 Mac IP：

ipconfig getifaddr en0
不要写 127.0.0.1：ESP32 里的这个地址表示 ESP32 自己，不是 Mac。

5. 抓一帧并作为二进制正文 POST

static void uploadFrame() {
    camera_fb_t* fb = esp_camera_fb_get();
    if (fb == nullptr) {
        Serial.println("upload: capture failed");
        return;
    }

    const uint32_t sequence = ++uploadSequence;
    const size_t frameLength = fb->len;

    HTTPClient http;
    http.setConnectTimeout(2000);
    http.setTimeout(5000);
    http.begin(kUploadUrl);
    http.addHeader("Content-Type", "image/jpeg");
    http.addHeader("X-Image-Sequence", String(sequence));

    const int status = http.POST(fb->buf, frameLength);
    esp_camera_fb_return(fb);

    String result;
    if (status > 0) {
        result = http.getString();
    } else {
        result = HTTPClient::errorToString(status);
    }
    http.end();

    Serial.printf("upload: seq=%lu bytes=%zu status=%d result=%s\n",
                  static_cast<unsigned long>(sequence), frameLength, status,
                  result.c_str());
}

POST() 返回后，HTTP 客户端已经使用完 JPEG 请求正文，所以可以立刻归还帧缓冲；读取服务器的小型 JSON 响应不再依赖 fb->buf。

6. 用 millis() 调度上传

static void uploadLoop() {
    if (!uploadEnabled || WiFi.status() != WL_CONNECTED || !cameraReady) {
        return;
    }

    const uint32_t now = millis();
    if (now - lastUploadAttemptMs < kUploadIntervalMs) {
        return;
    }

    lastUploadAttemptMs = now;
    uploadFrame();
}

把它加入主循环：

void loop() {
    handleSerialInput();
    heartbeatLoop();
    wifiLoop();
    httpLoop();
    uploadLoop();
}

时间戳在调用阻塞的 uploadFrame() 之前 更新。这样失败后不会每次 loop() 都立刻重试。

7. 用串口开关控制实验

在命令 switch 中加入：

case 'u':
    uploadEnabled = !uploadEnabled;
    lastUploadAttemptMs = millis();
    Serial.printf("upload: %s interval=%lu ms\n",
                  uploadEnabled ? "enabled" : "disabled",
                  static_cast<unsigned long>(kUploadIntervalMs));
    return;

按一次 u 后等待 10 秒，期望连续看到三组对应日志：

# ESP32
upload: seq=1 bytes=... status=201 result={"accepted": true, ...}

# Mac
image: seq=1 bytes=... jpeg=valid

8. 失败实验

  1. 保持上传开启，停止 Python server。
  2. 等待一次上传，记录负数状态和错误文本。
  3. 确认最多每 10 秒尝试一次，心跳能在请求超时后恢复。
  4. 重启 server，不重启 ESP32，确认后续上传自动恢复到 201。

本课不做立即重试、排队和图片缓存；这些属于第 13 课的可靠性策略。

9. 回忆检查

为什么必须在调用可能阻塞的 uploadFrame() 之前更新计时状态？请在答案中使用“尝试时间”“阻塞”“重试风暴”。

通过标准

  • Mac 自测 JPEG POST 返回 201。
  • ESP32 连续三次按约 10 秒间隔上传，Mac 均验证为完整 JPEG。
  • 服务器停止时出现有边界的失败，重启后自动恢复。
  • 上传成功和失败后，帧缓冲都已归还，心跳恢复。
