可靠性与可诊断性

第 13 课 · 预计 55 分钟 · 当前课程收尾

本课成果：设备能区分成功、可恢复失败和协议错误；网络、5xx、408 和 429 按 10→20→40→60 秒退避，成功后复位；确定的协议错误则停止自动上传。/health 暴露上传状态；服务端只保留最近 120 张并输出结构化日志。

1. 先确定失败策略

结果含义策略
201服务器已保存 JPEG成功计数 +1，退避复位到 10 秒
负数 / 5xx连接、超时或服务器暂时失败失败计数 +1，指数退避
408 / 429请求超时或服务端限流指数退避后重试
404/413/415/422路径、类型或正文违反本协议停止自动上传，修复后人工重新开启
摄像头失败没有拿到帧归类为可恢复失败并退避
本项目的有限缓存策略是 0 张待上传图片：失败帧立即归还，恢复后重新拍摄。这样断网不会耗尽 PSRAM。代价是断网期间的画面会丢失；若业务要求一张不丢，应使用 SD 卡上的有限持久队列，而不是长期占用摄像头缓冲。

2. 给设备增加明确状态

在上传全局状态旁加入：

constexpr uint32_t kMaxUploadBackoffMs = 60000;
constexpr int kCaptureFailed = -1000;

uint32_t currentUploadDelayMs = kUploadIntervalMs;
uint32_t uploadSuccessCount = 0;
uint32_t uploadFailureCount = 0;
uint8_t consecutiveUploadFailures = 0;
int lastUploadStatus = 0;

这些字段不是“多打印一些日志”，而是上传状态机的可观察状态。

3. 让单次上传返回结果

把 uploadFrame() 的返回类型改为 int，并替换为：

static int uploadFrame() {
    camera_fb_t* fb = esp_camera_fb_get();
    if (fb == nullptr) {
        Serial.println("{\"event\":\"upload\",\"error\":\"capture_failed\"}");
        return kCaptureFailed;
    }

    const uint32_t sequence = ++uploadSequence;
    const size_t frameLength = fb->len;

    HTTPClient http;
    http.setConnectTimeout(2000);
    http.setTimeout(5000);
    http.begin(kUploadUrl);
    http.addHeader("Content-Type", "image/jpeg");
    http.addHeader("X-Image-Sequence", String(sequence));

    const int status = http.POST(fb->buf, frameLength);
    esp_camera_fb_return(fb);
    http.end();

    Serial.printf(
        "{\"event\":\"upload\",\"seq\":%lu,\"bytes\":%zu,\"status\":%d}\n",
        static_cast<unsigned long>(sequence), frameLength, status);
    return status;
}

无论 HTTP 成功还是失败，POST() 返回后都归还帧缓冲。退避状态只保存数字，不保存那一大块 JPEG。

4. 把定时器升级成退避状态机

static bool isPermanentUploadFailure(int status) {
    return status >= 400 && status < 500
        && status != 408 && status != 429;
}

static void uploadLoop() {
    if (!uploadEnabled || WiFi.status() != WL_CONNECTED || !cameraReady) {
        return;
    }

    const uint32_t now = millis();
    if (now - lastUploadAttemptMs < currentUploadDelayMs) {
        return;
    }

    lastUploadAttemptMs = now;
    const int status = uploadFrame();
    lastUploadStatus = status;

    if (status == 201) {
        ++uploadSuccessCount;
        consecutiveUploadFailures = 0;
        currentUploadDelayMs = kUploadIntervalMs;
        return;
    }

    ++uploadFailureCount;
    if (consecutiveUploadFailures < 255) {
        ++consecutiveUploadFailures;
    }

    if (isPermanentUploadFailure(status)) {
        uploadEnabled = false;
        currentUploadDelayMs = kUploadIntervalMs;
        Serial.printf(
            "{\"event\":\"upload_disabled\",\"status\":%d,"
            "\"reason\":\"protocol_error\"}\n",
            status);
        return;
    }

    if (currentUploadDelayMs < kMaxUploadBackoffMs) {
        currentUploadDelayMs *= 2;
        if (currentUploadDelayMs > kMaxUploadBackoffMs) {
            currentUploadDelayMs = kMaxUploadBackoffMs;
        }
    }

    Serial.printf(
        "{\"event\":\"upload_backoff\",\"failures\":%u,"
        "\"next_delay_ms\":%lu}\n",
        static_cast<unsigned>(consecutiveUploadFailures),
        static_cast<unsigned long>(currentUploadDelayMs));
}

这里退避的是下一次新抓拍。网络断开时不会抓帧；Wi‑Fi 重连后，若等待时间已经满足，就会立即进行下一次尝试。

5. 人工重新开启时复位状态

把串口命令 u 改为：

case 'u':
    uploadEnabled = !uploadEnabled;
    consecutiveUploadFailures = 0;
    currentUploadDelayMs = kUploadIntervalMs;
    lastUploadAttemptMs = millis();
    Serial.printf("upload: %s interval=%lu ms\n",
                  uploadEnabled ? "enabled" : "disabled",
                  static_cast<unsigned long>(currentUploadDelayMs));
    return;

顺手修复现有的命令贯穿：在 case 'r' 的 WiFi.reconnect() 后加入 return;，否则它会继续进入 case 'p'，意外发送事件。

6. 让 /health 回答“系统现在怎样”

在 handleHealth() 的正文中追加：

const uint32_t elapsed = millis() - lastUploadAttemptMs;
const uint32_t nextInMs = !uploadEnabled || elapsed >= currentUploadDelayMs
    ? 0
    : currentUploadDelayMs - elapsed;

body += "upload_enabled = " + String(uploadEnabled ? "yes" : "no") + "\n";
body += "upload_successes = " + String(uploadSuccessCount) + "\n";
body += "upload_failures = " + String(uploadFailureCount) + "\n";
body += "upload_consecutive_failures = " + String(consecutiveUploadFailures) + "\n";
body += "upload_last_status = " + String(lastUploadStatus) + "\n";
body += "upload_next_in_ms = " + String(nextInMs) + "\n";

把原来的 body.reserve(96) 增大为 body.reserve(320)。现在无需翻串口历史，也能从浏览器检查状态。

7. 给服务端加有限保留

在 server.py 的 import 和常量区加入：

import threading

MAX_STORED_IMAGES = 120
STORAGE_LOCK = threading.Lock()


def prune_images_locked():
    files = sorted(IMAGE_DIR.glob("*.jpg"))
    expired = files[:-MAX_STORED_IMAGES]
    for path in expired:
        path.unlink(missing_ok=True)
    return len(expired)


def log_event(event, **fields):
    record = {
        "time": datetime.now(timezone.utc).isoformat(),
        "event": event,
        **fields,
    }
    print(json.dumps(record, ensure_ascii=False, separators=(",", ":")), flush=True)

写完临时文件后，先在锁内发布最终文件；保存失败返回 500：

try:
    with STORAGE_LOCK:
        os.replace(temporary_path, final_path)
except OSError as error:
    temporary_path.unlink(missing_ok=True)
    log_event("image_save_failed", error=str(error))
    self.send_json(500, {"error": "save failed"})
    return

图片已经发布成功后再做裁剪。裁剪失败需要记录，但不能谎称刚才的保存失败：

pruned = 0
try:
    with STORAGE_LOCK:
        pruned = prune_images_locked()
except OSError as error:
    log_event("image_prune_failed", error=str(error))

log_event(
    "image_saved",
    sequence=sequence,
    filename=filename,
    bytes=len(image),
    pruned=pruned,
)

删除原来的普通 print(f"image: ...")。一个日志事件占一行 JSON，之后可以稳定地被程序解析。

8. 增加服务端状态接口

在 do_GET() 处理 /api/images 之前加入：

if path == "/api/status":
    with STORAGE_LOCK:
        files = list(IMAGE_DIR.glob("*.jpg"))
        stored_bytes = sum(file.stat().st_size for file in files)
    self.send_json(200, {
        "stored_images": len(files),
        "stored_bytes": stored_bytes,
        "max_stored_images": MAX_STORED_IMAGES,
    })
    return

把 /api/images 的列表读取改成：

with STORAGE_LOCK:
    files = sorted(IMAGE_DIR.glob("*.jpg"))[-MAX_LISTED_IMAGES:]
    images = [
        {"name": file.name, "url": f"/images/{file.name}"}
        for file in files
    ]

在 /images/... 分支中也应在同一锁内完成 is_file() 和 read_bytes()，然后释放锁再调用 send_bytes()。这样裁剪线程不会恰好在检查与读取之间删除文件。

9. 故障注入矩阵

实验操作必须看到
正常连续成功三次status=201，失败连续数归零，间隔回到 10 秒
服务不可达停止 Python server负数状态；下一间隔 20→40→60 秒，不忙循环
自动恢复重新启动 server下一次成功后退避复位为 10 秒
协议错误临时把路径改成 /wrong404 后自动上传关闭，不再重复请求
有限磁盘临时把上限改为 5，上传至少 8 张目录始终只有最新 5 张，日志出现 pruned

10. 回忆检查

为什么网络错误适合指数退避，而 404 不应该无限重试？请在答案中使用“暂时故障”“协议错误”“资源上限”。

通过标准

  • 成功、暂时失败、408/429 和确定的协议错误分别进入正确策略。
  • 停服后退避增长，恢复后自动归零。
  • /health 能解释当前上传状态和下一次等待。
  • 服务端文件数有上限，结构化日志可逐行解析。
  • 所有失败路径都归还摄像头帧，设备持续心跳且不会耗尽内存。
