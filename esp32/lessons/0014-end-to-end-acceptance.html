端到端验收

第 14 课 · 预计 30–45 分钟 · 不增加新功能

本课成果：用可重复证据证明使命已经完成：设备可启动、抓拍、联网、定时上传、保存、播放，并能从服务不可达、协议错误和存储边界中恢复或安全停用。

当前真实起点

开课检查时，ESP32 的 /health 返回：

upload_enabled = yes
upload_successes = 18
upload_failures = 4
upload_consecutive_failures = 2
upload_last_status = -1
upload_next_in_ms ≈ 38581

同时 Mac 的 127.0.0.1:8000 不可连接。这已经证明“服务不可达 → 负数状态 → 退避”正在真实设备上运行。第一项验收就是在不重启 ESP32 的情况下恢复 server。

1. 恢复链路

在 Mac 新终端启动服务：

cd /Users/watt/Code/Teach/esp32/esp32-test
python3 server/server.py

等待最多 60 秒，再检查：

curl http://192.168.1.195/health
curl http://127.0.0.1:8000/api/status

通过证据：upload_successes 增加、upload_consecutive_failures=0、upload_last_status=201，并且下一间隔恢复约 10 秒。

2. 验收单张图片链路

curl -D /tmp/capture.headers \
  http://192.168.1.195/capture \
  -o /tmp/capture.jpg
cat /tmp/capture.headers
file /tmp/capture.jpg

必须看到 200、Content-Type: image/jpeg、非零长度，并被 file 识别为 JPEG。

3. 验收保存与播放

curl http://127.0.0.1:8000/api/images | python3 -m json.tool
curl http://127.0.0.1:8000/api/status | python3 -m json.tool
find server/images -type f -name '*.tmp'

打开 时间相册，等待列表刷新并检查上一张、暂停、下一张。临时文件查询应无输出。

4. 五分钟稳定性观察

先保存一次 health，五分钟后再次保存：

curl http://192.168.1.195/health -o /tmp/health-before.txt
# 等待五分钟
curl http://192.168.1.195/health -o /tmp/health-after.txt
diff -u /tmp/health-before.txt /tmp/health-after.txt

预期成功数持续增加、连续失败为 0；free heap 可以波动，但不应每次采样都单向下降。串口心跳持续，设备没有重启。

5. 协议错误应安全停止

临时把 kUploadUrl 的路径改成 /wrong，烧录并按 u 开启。看到一次 404 后，health 应显示：

upload_enabled = no
upload_last_status = 404

确认不会继续重复请求。然后恢复 /images、重新烧录并按 u。

6. 验收服务端容量边界

临时把 MAX_STORED_IMAGES 改成 5，重启 server，产生至少 8 次成功上传：

find server/images -type f -name '*.jpg' | wc -l
curl http://127.0.0.1:8000/api/status

文件数应稳定为 5，结构化日志中出现 "pruned":1。完成后恢复为 120 并重启 server。

7. 从内存画出完整系统

不要看前面的代码，用自己的话解释下面五个边界。答案应包含“所有权”“FreeRTOS”“原子发布”“指数退避”。
