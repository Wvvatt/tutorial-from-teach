运行时修改上传间隔

第 15 课 · 预计 30 分钟 · 进阶阶段

本课成果：完整功能已经写入 main.cpp。烧录一次后，无需再次编译便可通过 GET /config 查看上传间隔，通过 POST /config?interval_ms=... 修改它；设备会拒绝缺失、非数字和越界配置。

1. 先分清两种配置

constexpr 会在编译时固定下来；普通变量是设备运行期间的状态。最终代码把默认值和当前值明确分开：

constexpr uint32_t kDefaultUploadIntervalMs = 10000;
constexpr uint32_t kMinUploadIntervalMs = 1000;
constexpr uint32_t kMaxUploadIntervalMs = 300000;
uint32_t uploadIntervalMs = kDefaultUploadIntervalMs;

uploadIntervalMs 是当前正常上传间隔；kMaxUploadBackoffMs 仍是失败重试策略的边界，两者不能合并。

这一课不保存到 Flash：重启后恢复 10 秒是预期行为。运行时配置与断电持久化是两个不同问题。

2. 读取接口

完整实现用 JSON 返回当前状态：

static void handleGetConfig() {
    String body = "{\"interval_ms\":" + String(uploadIntervalMs) + "}\n";
    server.send(200, "application/json", body);
}

GET 只观察状态，不改变状态。

3. 解析、校验，再提交

解析器要求每个字符都是数字，再检查范围；这样空字符串、负数、空格和尾随字符都不能混过去：

static bool parseUploadInterval(const String& raw,
                                uint32_t& parsedIntervalMs) {
    if (raw.isEmpty()) {
        return false;
    }

    for (size_t index = 0; index < raw.length(); ++index) {
        if (raw[index] < '0' || raw[index] > '9') {
            return false;
        }
    }

    char* end = nullptr;
    const unsigned long parsed = std::strtoul(raw.c_str(), &end, 10);
    if (*end != '\0' || parsed < kMinUploadIntervalMs
        || parsed > kMaxUploadIntervalMs) {
        return false;
    }

    parsedIntervalMs = static_cast<uint32_t>(parsed);
    return true;
}

static void handlePostConfig() {
    if (!server.hasArg("interval_ms")) {
        server.send(400, "text/plain; charset=utf-8",
                    "missing interval_ms\n");
        return;
    }

    uint32_t parsedIntervalMs = 0;
    if (!parseUploadInterval(server.arg("interval_ms"),
                             parsedIntervalMs)) {
        server.send(400, "text/plain; charset=utf-8",
                    "interval_ms must be an integer from 1000 to 300000\n");
        return;
    }

    uploadIntervalMs = parsedIntervalMs;
    currentUploadDelayMs = uploadIntervalMs;
    lastUploadAttemptMs = millis();

    Serial.printf("config: upload interval=%lu ms\n",
                  static_cast<unsigned long>(uploadIntervalMs));
    handleGetConfig();
}

关键顺序是解析 → 校验 → 提交。非法输入不能先写入真实状态。更新时间边界后，下一次上传从本次配置提交时重新计时。

4. 同一路径的两种方法

最终代码在 httpLoop() 的一次性注册区包含：

server.on("/config", HTTP_GET, handleGetConfig);
server.on("/config", HTTP_POST, handlePostConfig);

路径相同，但 HTTP method 不同，因此会进入不同 handler。

6. 故意输入坏数据

curl -i -X POST http://192.168.1.195/config
curl -i -X POST 'http://192.168.1.195/config?interval_ms=hello'
curl -i -X POST 'http://192.168.1.195/config?interval_ms=100'
curl http://192.168.1.195/config

前三个请求都必须返回 400，最后仍应是 30000。错误请求不应污染旧配置。

7. 回忆检查

为什么 handlePostConfig() 修改 uploadIntervalMs 时暂时不需要 mutex？请在答案中使用“同一个 loop”“顺序执行”“共享状态”。

通过标准

  • GET /config 返回当前上传间隔。
  • 合法 POST 返回 200，无需烧录便改变上传节奏。
  • 缺失、非数字和越界参数返回 400，旧值保持不变。
  • 能解释为什么当前没有数据竞争，以及将 handler 放到独立 FreeRTOS task 后为什么必须重新评估。
