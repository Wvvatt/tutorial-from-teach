V4L2 架构与 Two-Queue 模型

理解 V4L2 decoder 的骨架——两个队列、四种 buffer 模式、以及 decoder 在其中的定位。

→ Lesson 2: Stateful Decoder 状态机

1. V4L2 是什么

Video for Linux 2 (V4L2) 是 Linux 内核中视频设备的用户空间 API 规范与驱动框架。它统一了以下设备的编程接口：

  • 📷 摄像头（Video Capture）
  • 🖥️ 视频输出（Video Output）
  • 🎬 视频编解码器（Memory-to-Memory Codec）—— 你的关注点
  • 📻 Radio / SDR
  • ✋ VBI / Touch / Metadata

V4L2 设备节点通常为 /dev/videoN（如 /dev/video0），通过标准的 open / close / ioctl 系统调用进行控制。

→ 见 V4L2 设备节点命名规则

2. Two-Queue 模型——V4L2 的基石

对于 memory-to-memory 设备（decoder / encoder），V4L2 的核心抽象是 两个独立的 buffer 队列：

OUTPUT Queue ──────────────────────────► CAPTURE Queue
用户填入编码码流 从硬件读出解码帧
V4L2_BUF_TYPE_VIDEO_OUTPUT V4L2_BUF_TYPE_VIDEO_CAPTURE

┌───────────────────────────────────────────────────────┐
│ Decoder 硬件/驱动 │
│ OUTPUT: 编码码流 in → [H.264/HEVC/VP9...] → 原始帧 out : CAPTURE │
└───────────────────────────────────────────────────────┘
  OUTPUT (Source) CAPTURE (Destination)
对 decoder 编码码流（如 H.264 ES） 解码后原始帧（如 NV12）
Buffer 类型 VIDEO_OUTPUT / _MPLANE VIDEO_CAPTURE / _MPLANE
数据方向 用户 → 驱动 驱动 → 用户
关键洞察：OUTPUT 和 CAPTURE 两个 queue 独立运行。一个 OUTPUT buffer 可能产生 0 个、1 个或多个 CAPTURE buffer（取决于帧重排、帧类型等）。

3. 基本 I/O 生命周期

无论 stateful 还是 stateless，所有 V4L2 decoder 都遵循以下基本步骤：

1. open("/dev/videoN")
2. VIDIOC_QUERYCAP        // 查询设备能力
3. VIDIOC_S_FMT(OUTPUT)   // 设置编码格式
4. VIDIOC_REQBUFS(OUTPUT) // 分配 OUTPUT buffer
5. VIDIOC_STREAMON(OUTPUT) // 启动 OUTPUT 队列
6. VIDIOC_G_FMT(CAPTURE)  // 获取解码帧格式
7. VIDIOC_REQBUFS(CAPTURE)// 分配 CAPTURE buffer
8. VIDIOC_STREAMON(CAPTURE)// 启动 CAPTURE 队列
   ——— 解码循环 ———
9.   VIDIOC_QBUF(OUTPUT)  // 送入码流
10.  VIDIOC_DQBUF(CAPTURE)// 取出解码帧
   ——— 结束 ———
11. VIDIOC_STREAMOFF(both)
12. close(fd)

这个流程在 stateful decoder 中基本如此；stateless decoder 则在此基础上增加了 Request API 来提交解码参数（见后续课程）。

4. Stateful vs Stateless —— 最根本的区别

工作中接触到 decoder 时，第一件事就是分清你面对的是 stateful 还是 stateless 接口。它们的核心差异：

维度 Stateful Decoder Stateless Decoder
谁维护解码状态 驱动/硬件 用户空间
输入 完整码流（Annex-B / raw stream） 每帧码流 + 完整的解码参数（SPS/PPS/参考帧列表等）
使用 Request API 否 是（通过 MEDIA_REQUEST_IOC_QUEUE）
状态机复杂度 复杂（初始化→解码→DRC→drain→seek） 简单（每帧独立提交）
典型硬件 传统 ASIC decoder 现代 GPU/VPU（如 Qualcomm venus, Rockchip vdec）
何时用哪个？如果硬件内部有完整的解码管道（parser + DPB 管理），用 stateful 接口。如果硬件只做纯粹的熵解码/MC，解码状态管理交给用户空间，用 stateless 接口 + Request API。

5. Buffer 的四种内存模式

V4L2 定义了四种 buffer 内存管理模式，选择哪一种取决于系统架构和性能需求：

  • V4L2_MEMORY_MMAP：驱动分配内存，用户通过 mmap 映射访问。最常用，性能好。
  • V4L2_MEMORY_USERPTR：用户分配内存，驱动通过指针访问。适合已有 buffer pool 的应用。
  • V4L2_MEMORY_DMABUF：通过 dma-buf fd 共享 buffer。零拷贝路径（decoder → display）。
  • V4L2_MEMORY_OVERLAY：直接将视频覆盖到 framebuffer（已废弃）。

对于 decoder，CAPTURE 队列通常使用 MMAP 或 DMABUF（后者用于零拷贝显示流水线）。

7. 动手：查找系统中的 V4L2 设备

即使你在 macOS 上没有真实的 V4L2 硬件，你仍然可以了解 V4L2 设备节点在 Linux 上长什么样：

# 在 Linux 上查看所有 V4L2 设备
ls -la /dev/video*

# 查看设备能力
v4l2-ctl --device=/dev/video0 --all

# 枚举 decoder 支持的格式
v4l2-ctl --device=/dev/video0 --list-formats
v4l2-ctl --device=/dev/video0 --list-formats-out

如果需要在 macOS 上实验，可以考虑：

  • 使用 QEMU 模拟 Linux 环境
  • 加载 vivid 虚拟测试驱动（modprobe vivid）
