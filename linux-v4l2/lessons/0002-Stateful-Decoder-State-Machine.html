Stateful Decoder 状态机详解

从初始化到 Drain——理解 stateful V4L2 decoder 的完整生命周期。

← Lesson 1: V4L2 概览  |  → Lesson 3: Stateless Decoder & Request API

1. 状态机全景图

Stateful decoder 在 GLOSSARY 中的定义是：由驱动/硬件维护解码状态（DPB、参考帧列表等），用户只需送入完整码流，按 display order 取出解码帧。它的行为由一个 状态机 精确控制。

UNINIT
↓ S_FMT(OUTPUT) + REQBUFS + STREAMON
INIT
↓ 解析到流分辨率 → SOURCE_CHANGE 事件
CAPTURE_SETUP
↓ G_FMT + REQBUFS + STREAMON(CAPTURE)
DECODING
↕ QBUF / DQBUF 循环
┌─────────────────┤
DRC/源改变
├─────────────────┐
Drain/Flush
├──────────┐
EOS
DRC → CAPTURE_SETUP
DRAIN → DECODING
EOS → (停止)

→ 基于内核 Stateful Decoder 文档 中的 DOT 状态图

核心原则：OUTPUT queue 是"主人"——先设置 OUTPUT 格式，CAPTURE 格式由此衍生。在 buffer 已分配时不能更改 OUTPUT 格式（返回 -EBUSY）。

2. 阶段一：初始化（UNINIT → INIT → CAPTURE_SETUP）

Decoder 从无到有的过程分为几步，每一步都有精确的 ioctl 序列：

2.1 设置 OUTPUT 格式

struct v4l2_format fmt = {
    .type = V4L2_BUF_TYPE_VIDEO_OUTPUT_MPLANE,
    .fmt.pix_mp = {
        .width        = 1920,
        .height       = 1088,       // coded resolution（宏块对齐）
        .pixelformat  = V4L2_PIX_FMT_H264,  // 编码格式
        .sizeimage    = 1024 * 1024,         // OUTPUT buffer 大小
    },
};
ioctl(fd, VIDIOC_S_FMT, &fmt);

调用 VIDIOC_S_FMT(OUTPUT) 后，驱动会：

  • 根据你设置的 coded format 更新 sizeimage 等字段
  • 初步推算 CAPTURE 格式（但还不是真实分辨率）
  • 如果码流自带分辨率信息，真正的 CAPTURE 格式要等解析到流之后才确定

2.2 分配 OUTPUT buffer 并启动

struct v4l2_requestbuffers req = {
    .count  = 4,
    .type   = V4L2_BUF_TYPE_VIDEO_OUTPUT_MPLANE,
    .memory = V4L2_MEMORY_MMAP,
};
ioctl(fd, VIDIOC_REQBUFS, &req);      // 分配 4 个 OUTPUT buffer
ioctl(fd, VIDIOC_STREAMON, &req.type);  // 启动 OUTPUT 队列

2.3 送入码流直到解析到分辨率

对于自带分辨率信息的码流（如 H.264），你需要：

  • QBUF 送入码流数据 → 驱动解析 SPS/PPS/IDR
  • 驱动解析出真实分辨率后，发出 V4L2_EVENT_SOURCE_CHANGE 事件（V4L2_EVENT_SRC_CH_RESOLUTION）
  • 收到该事件前，G_FMT(CAPTURE) 返回的可能是占位值
关键细节：在 INIT 阶段送入码流但还没解析到分辨率时，已送入的 buffer 不会返回 decoded frame——因为还没配置 CAPTURE queue。"no decoded frames are produced during this phase"（内核文档原话）。

3. 阶段二：解码（DECODING）

CAPTURE queue 配置完成后，进入正常的解码循环。但 OUTPUT 和 CAPTURE 的关系 不是 一一对应的：

⚠️
一个 OUTPUT buffer 可能产生 0、1 或多个 CAPTURE buffer。
• 只有元数据的 buffer → 0 个 CAPTURE
• 包含一帧数据 → 1 个 CAPTURE（最常见）
• 包含多帧数据 → 多个 CAPTURE
• 帧重排 → CAPTURE 的 display order ≠ OUTPUT 的 decode order

为了匹配 OUTPUT 和 CAPTURE buffer，使用 timestamp 字段：

// QBUF 时设置 timestamp
struct v4l2_buffer buf = {
    .type      = V4L2_BUF_TYPE_VIDEO_OUTPUT_MPLANE,
    .index     = 0,
    .timestamp = { .tv_sec = seq, .tv_usec = 0 },  // 用户自定义
};
ioctl(fd, VIDIOC_QBUF, &buf);

// DQBUF 时 CAPTURE buffer 的 timestamp 会被设成它来源的 OUTPUT buffer 的值
ioctl(fd, VIDIOC_DQBUF, &capture_buf);
// capture_buf.timestamp == 对应 OUTPUT 的 timestamp
⚠️ 关于 reference frame buffer 的重要警告：
被用作参考帧的 CAPTURE buffer 的内存 即使被 DQBUF 后，硬件仍然可能在读取它。在 CAPTURE streaming 期间不要写入已 dequeue 的 CAPTURE buffer 内存，否则会导致后续帧解码错误。

4. 阶段三：Seek（跳转）

Seek 由 OUTPUT queue 控制：

  1. STREAMOFF(OUTPUT) — 停止 OUTPUT，所有 pending 的 OUTPUT buffer 被丢弃并返回
  2. STREAMON(OUTPUT) — 重新启动 OUTPUT，开始接受新数据
  3. QBUF 新位置的码流 — 驱动会自动寻找 resume point（关键帧 / SPS/PPS/IDR）
  4. 找到 resume point 后，开始产生解码帧
注意：STREAMOFF 后，CAPTURE queue 可能还残留之前解码的帧。如果你想要"瞬时 seek"，可以对 CAPTURE 也做 STREAMOFF/STREAMON 来丢弃它们。

5. 阶段四：Dynamic Resolution Change（DRC）

如果码流中分辨率发生变化（这在自适应流中很常见），decoder 会触发 DRC 序列。注意：不是所有 decoder 都支持 DRC——需要在 ENUM_FMT 时检查 V4L2_FMT_FLAG_DYN_RESOLUTION 标志。

DRC 的触发条件（内核文档列出）：

  • coded resolution 变化
  • visible resolution 变化
  • 所需最小 buffer 数量变化
  • bit-depth 变化
  • colorspace 变化

DRC 的处理流程：

  1. 驱动发出 V4L2_EVENT_SOURCE_CHANGE（V4L2_EVENT_SRC_CH_RESOLUTION）
  2. 驱动解码完所有变化前的帧 — 最后一个 CAPTURE buffer 标记 V4L2_BUF_FLAG_LAST
  3. 再次执行 CAPTURE_SETUP — G_FMT → (可选 REQBUFS) → STREAMON
  4. 恢复解码
重要：DRC 期间 OUTPUT queue 必须保持 streaming！如果对 OUTPUT 做 STREAMOFF，就会触发 seek 而不是 DRC。

6. 阶段五：Drain（排空）

当你想确保所有已送入的码流都被完全解码（比如切换码流/关闭前），需要 drain decoder：

  1. 发送 V4L2_DEC_CMD_STOP — ioctl(fd, VIDIOC_DECODER_CMD, &(struct v4l2_decoder_cmd){.cmd = V4L2_DEC_CMD_STOP})
  2. 继续处理两边的 buffer — OUTPUT buffer 继续被处理并 DQBUF；CAPTURE buffer 继续被产出并 DQBUF
  3. 等待 V4L2_BUF_FLAG_LAST — 最后一个 CAPTURE buffer 会带有此标志，表示 drain 完成
  4. 恢复解码 — 发送 V4L2_DEC_CMD_START

Drain 完成后，decoder 状态不变（继续解码后续数据），但如果没有后续数据了就是 EOS。

8. 小测验

1. 初始化阶段，decoder 如何知道真正的流分辨率？

A. 从 S_FMT(OUTPUT) 的 width/height 参数获得
B. 解析码流后通过 V4L2_EVENT_SOURCE_CHANGE 通知用户
C. 从 VIDIOC_QUERYCAP 中读取
D. 从 VIDIOC_ENUM_FMT 返回值中获得

2. DRC（Dynamic Resolution Change）发生时，OUTPUT queue 应该怎么做？

A. 保持 streaming，继续送入码流
B. 立即 STREAMOFF，等 CAPTURE 重新配置好再重启
C. 先 drain 再重新初始化
D. 关闭文件描述符重新 open

3. Drain 完成的标志是什么？

A. 所有 OUTPUT buffer 被 DQBUF 返回
B. 收到 V4L2_EVENT_EOS
C. OUTPUT 队列为空
D. 最后一个 CAPTURE buffer 带有 V4L2_BUF_FLAG_LAST 标志

4. Decoder 正常解码时，一个 OUTPUT buffer 一定产生一个 CAPTURE buffer 吗？

A. 是的，一一对应
B. 不一定，可能 0 个、1 个或多个
C. 至少产生一个 CAPTURE buffer
D. 取决于 CAPTURE buffer 数量是否足够
