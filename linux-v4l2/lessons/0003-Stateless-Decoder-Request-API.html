Lesson 3

Stateless Decoder 与 Request API

理解每帧独立提交模型，以及 Request API 如何将 buffer 和 controls 原子绑定
← Lesson 2: Stateful Decoder 状态机  |  → Lesson 4: V4L2 调试工具实战

0. 快速回顾：Stateful 模型做了什么

在 Lesson 2 中，我们看到 stateful decoder 的驱动维护了所有解码状态：

驱动负责用户负责
维护 DPB、参考帧列表、解码顺序送入完整码流
自动查找 resume point按 display order 取出帧
管理 buffer 生命周期（哪些是参考帧）不写入已 dequeue 的 buffer
处理 SPS/PPS 解析发送 DEC_CMD_STOP 来 drain

Stateless decoder 正好相反：驱动不做上述任何事，所有状态维护都在用户空间。

核心哲学：驱动是一个"纯变换函数"——给一组 buffer + 一组 controls，产出解码帧。驱动不在帧之间保留任何状态。

1. 什么是 Stateless Decoder

Stateless decoder 工作方式的核心类比：

Stateful ≈ 你把整本书交给翻译，他按章节顺序交出译文。
Stateless ≈ 你把每一页单独交给翻译，同时告诉他"这一页是第 3 章第 2 节，参考了第 5 页和第 8 页的数据"。翻译自己什么都不记。

在 V4L2 的语境下，这意味着：

  • 用户必须自己解析码流中的 SPS/PPS（H.264）或 frame header（VP8/VP9）
  • 用户必须自己管理 DPB：跟踪哪些 buffer 是参考帧，哪些 buffer 被哪些帧引用
  • 用户必须为每帧提供完整的解码参数，通过 V4L2 controls 传递
  • 解码参数和输入数据必须 原子提交，这正是 Request API 的用途

2. Request API 基础

Request API 的核心思想：将一组 V4L2 操作打包成一个原子单元，一起提交给驱动。

Request 的生命周期

ALLOC → PREPARE
QBUF + S_EXT_CTRLS → QUEUE
提交给驱动 → COMPLETE
poll() 等待 → REINIT / close

关键系统调用

操作系统调用说明
分配 Request ioctl(media_fd, MEDIA_IOC_REQUEST_ALLOC, &req_fd) 从 media 设备分配一个 request，返回 FD
绑定 buffer 到 Request QBUF 时设置 buf.flags |= V4L2_BUF_FLAG_REQUEST_FD
并设置 buf.request_fd = req_fd		 告诉驱动：这个 buffer 属于指定 request
绑定 controls 到 Request S_EXT_CTRLS 时设置 which = V4L2_CTRL_WHICH_REQUEST_VAL
并设置 request_fd = req_fd		 控件的值在 request 提交时才生效
提交 Request ioctl(req_fd, MEDIA_REQUEST_IOC_QUEUE) 原子提交：buffer 开始被处理，controls 一次性应用
等待完成 poll(&(struct pollfd){.fd=req_fd, .events=POLLPRI}, 1, -1) 阻塞等待 request 完成
回收 Request ioctl(req_fd, MEDIA_REQUEST_IOC_REINIT) 清空状态，复用该 request

3. Stateless Decoder 的完整流程

3.1 初始化

  1. S_FMT(OUTPUT) — 设置 coded format（如 V4L2_PIX_FMT_H264_SLICE）及其宽高
  2. S_EXT_CTRLS — 设置 codec 级别的初始参数（如 H.264 profile/level，或初始的 SPS/PPS）
  3. G_FMT(CAPTURE) — 获取驱动推荐的 CAPTURE 格式
  4. [可选] S_FMT(CAPTURE) — 选择不同的 raw format
  5. REQBUFS(OUTPUT) — 分配源 buffer（bytestream 数据存放于此）
  6. REQBUFS(CAPTURE) — 分配目标 buffer（解码帧）
    注意：USERPTR 不兼容 CAPTURE queue
  7. MEDIA_IOC_REQUEST_ALLOC — 为每个 OUTPUT buffer 分配一个 request
  8. STREAMON(OUTPUT) + STREAMON(CAPTURE)
对比 Stateful：stateless 没有 INIT → CAPTURE_SETUP → DECODING 的状态变换。初始化完成后直接进入解码循环，状态全由用户管理。没有 V4L2_EVENT_SOURCE_CHANGE，没有自动等待 resolve。

3.2 每帧解码的循环

这是 stateless decoder 最关键的操作——每帧独立提交：

// 对于每一帧：
// 1. 准备 OUTPUT buffer（bytestream 数据）
struct v4l2_buffer out_buf = {
    .type       = V4L2_BUF_TYPE_VIDEO_OUTPUT_MPLANE,
    .index      = out_idx,
    .bytesused  = frame_data_size,
    .flags      = V4L2_BUF_FLAG_REQUEST_FD,
    .request_fd = req_fd,
    .timestamp  = { .tv_sec = frame_num, .tv_usec = 0 },
};
ioctl(decoder_fd, VIDIOC_QBUF, &out_buf);

// 2. 设置解码参数（H.264 示例）
struct v4l2_ext_controls ctrls = {
    .which      = V4L2_CTRL_WHICH_REQUEST_VAL,
    .request_fd = req_fd,
    .count      = /* 多少个 controls */,
    .controls   = /* struct v4l2_ext_control 数组 */,
};
// controls 数组包含：
//   - V4L2_CID_STATELESS_H264_SPS
//   - V4L2_CID_STATELESS_H264_PPS
//   - V4L2_CID_STATELESS_H264_SLICE_PARAMS
//   - V4L2_CID_STATELESS_H264_DECODE_PARAMS (=DPB 引用信息)
//   - V4L2_CID_STATELESS_H264_SCALING_MATRIX (可选)
ioctl(decoder_fd, VIDIOC_S_EXT_CTRLS, &ctrls);

// 3. 提交 request（原子起效）
ioctl(req_fd, MEDIA_REQUEST_IOC_QUEUE);

// 4. 等待 CAPTURE 产出
ioctl(decoder_fd, VIDIOC_DQBUF, &capture_buf);
// capture_buf.timestamp == frame_num（从 OUTPUT 传播而来）
📌
QBUF 和 S_EXT_CTRLS 的顺序可以互换。控件在 request 提交时的最终状态才生效。可以分多次 S_EXT_CTRLS 调用，也可以覆盖之前设置的值。

4. HOLD_CAPTURE_BUF：多 Slice 帧的处理

H.264 等格式支持一帧被分割成多个 slice。每个 slice 可以作为一个独立的 OUTPUT buffer 提交，但它们属于同一帧。

此时需要 V4L2_BUF_FLAG_M2M_HOLD_CAPTURE_BUF 标志：

// Slice 1（非最后一个 slice）
out_buf.flags |= V4L2_BUF_FLAG_REQUEST_FD
              |  V4L2_BUF_FLAG_M2M_HOLD_CAPTURE_BUF;
out_buf.timestamp = 42;  // 帧的 unique ID

// Slice 2（也是非最后一个）
out_buf.flags |= V4L2_BUF_FLAG_REQUEST_FD
              |  V4L2_BUF_FLAG_M2M_HOLD_CAPTURE_BUF;
out_buf.timestamp = 42;  // 同一个 timestamp！

// Slice 3（最后一个 slice——不设 HOLD）
out_buf.flags |= V4L2_BUF_FLAG_REQUEST_FD;
// 不设 HOLD_CAPTURE_BUF！
out_buf.timestamp = 42;  // 同一个 timestamp
规则：相同 timestamp 的多个 OUTPUT buffer 会解码到同一个 CAPTURE buffer。
HOLD_CAPTURE_BUF 告诉驱动："这个 CAPTURE buffer 还没完成，先别让它出队"。当最后一个 slice 的 OUTPUT buffer 不带 HOLD 且 timestamp 发生变化时，CAPTURE buffer 才会被完成并可供 DQBUF。
不是所有硬件都支持多 slice！只有 ENUM_FMT 时返回 V4L2_BUF_CAP_SUPPORTS_M2M_HOLD_CAPTURE_BUF 的驱动才支持。如果不支持，你需要自己在用户空间把同一帧的所有 slice 拼成一个 buffer 再提交。

5. Buffer 管理：用户空间的 DPB

Stateless decoder 最大的复杂度来自 用户空间必须管理 DPB（Decoded Picture Buffer）。

核心规则

  1. CAPTURE buffer 出队后，用户可以使用它（显示/合成），但 如果它是参考帧，不能马上重新入队
  2. 只有当引用该 buffer 的所有帧都解码完毕后，才能把它重新放回 CAPTURE queue
  3. 通过 timestamp 来引用参考帧：将 OUTPUT buffer 的 timestamp 转换为纳秒，填入 decode params 的参考帧列表
// 引用参考帧的典型做法
struct v4l2_ctrl_h264_decode_params dp = { /* ... */ };

// 参考帧列表中的每个条目引用一个之前解码帧的 timestamp（转纳秒）
dp.dpb[0] = v4l2_timeval_to_ns(&ref_frame_0_timestamp);
dp.dpb[1] = v4l2_timeval_to_ns(&ref_frame_1_timestamp);

// 设置 decode params control
struct v4l2_ext_control ctrl = {
    .id    = V4L2_CID_STATELESS_H264_DECODE_PARAMS,
    .ptr   = &dp,
    .size  = sizeof(dp),
};
🔑
安全做法：在所有引用某参考帧的帧都 DQBUF 之前，不要把该参考帧 buffer 重新 QBUF 到 CAPTURE queue。

进阶做法：如果驱动保证 CAPTURE buffer 按入队顺序处理（FIFO），且你已确保引用该参考帧的所有请求都已提交 + 数量足够的 CAPTURE buffer 已入队，则可以提前重新入队参考帧。

引用计数保护

当你通过 Request 引用一个参考帧时，驱动会增加该帧对应资源（如 DMABUF）的引用计数。这意味着：即使你关闭了参考帧 DMABUF 的 FD，内核也不会释放它的内存，直到引用它的所有 request 都完成。

6. Stateful vs Stateless — 完整对比

维度Stateful DecoderStateless Decoder
解码状态 驱动维护（DPB、参考帧列表、解码顺序） 用户空间维护全部状态
码流输入 原始 bytestream，驱动负责解析 SPS/PPS 用户需解析码流头，以 controls 形式传入
每帧操作 QBUF(OUTPUT) + DQBUF(CAPTURE)，无需额外参数 QBUF(OUTPUT) + S_EXT_CTRLS + QUEUE request
DPB 管理 驱动管理（用户只需不写入参考帧 buffer） 用户管理（追踪引用关系，安全回收 buffer）
每帧参数 驱动从码流中自动提取 用户通过 controls 显式提供所有参数
初始化 UNINIT → INIT → SOURCE_CHANGE → CAPTURE_SETUP → DECODING 直接初始化两队列 → 开始解码循环
DRC 驱动发出 V4L2_EVENT_SOURCE_CHANGE
用户重新配置 CAPTURE		 用户从码流中检测变化
STREAMOFF 两队列，重新初始化
Drain DEC_CMD_STOP → 等待 V4L2_BUF_FLAG_LAST DEC_CMD_FLUSH（如有 held buffer）
然后等待所有 request 完成
Seek STREAMOFF/ON(OUTPUT)，驱动找 resume point 直接提交新位置的帧 + 对应的 controls
pause 不送数据即可 不送数据即可 (更简单)
驱动复杂度 高 — 需实现 DPB、参考帧管理、码流解析 低 — 纯"帧变换"，无状态逻辑
用户复杂度 低 — 只需管理 buffer 生命周期 高 — 需实现完整的 DPB 管理器和码流解析器
典型应用 嵌入式播放器、简单解码应用 Chromium（VA-API）、Android Codec v2、GStreamer stateless plugin

7. 何时选择 Stateless？

Stateless 看起来"更难用"，但它存在的理由：

  • 硬件设计简单：解码器 ASIC 只需要实现纯解码运算，不需要内嵌码流解析器或 DPB 管理——这对 SoC 厂商来说是巨大优势
  • 用户空间有天然优势：码流解析和 DPB 管理在用户空间做更灵活（容易升级、跨平台共享代码）
  • 与图形栈集成：Chromium 等浏览器引擎已经有自己的码流解析器（用于软件解码），stateless 接口让它们直接复用
  • 原子提交模型：Request API 保证了 buffer 和 controls 同时提交，消除了竞态条件
  • 无 DRC 延迟：用户自己检测分辨率变化并重构，不需要等待驱动的事件通知
但也要注意：用户空间的码流解析器可能引入安全风险。如果解析器和硬件驱动对码流格式的理解不一致，可能导致越界访问或解码错误。内核文档明确说，驱动必须验证用户传入的所有参数。

8. 实操提示：如何判断一个 decoder 是 stateful 还是 stateless

# 方法 1：检查 CAPTURE queue 是否支持 REQUEST
v4l2-ctl -d /dev/video0 --list-formats-out

# 方法 2：检查 CAPTURE queue 的 capabilities 中是否有 V4L2_BUF_CAP_SUPPORTS_REQUESTS
v4l2-ctl -d /dev/video0 --get-fmt-video-out

# 方法 3：看 coded format 的名字
#   stateful coded format:   V4L2_PIX_FMT_H264     (不带 _SLICE)
#   stateless coded format:  V4L2_PIX_FMT_H264_SLICE
#   stateful coded format:   V4L2_PIX_FMT_VP8
#   stateless coded format:  V4L2_PIX_FMT_VP8_FRAME

# 方法 4：看设备 capability
#   stateful: V4L2_CAP_VIDEO_M2M (普通 mem2mem)
#   stateless: V4L2_CAP_VIDEO_M2M_MPLANE + BUF_CAP_SUPPORTS_REQUESTS

9. 小测验

1. Stateless decoder 中，谁负责管理 DPB（Decoded Picture Buffer）？

驱动维护，用户通过 VIDIOC_G_DPB 查询
用户空间维护，通过 controls 传递引用关系
硬件自动管理，无需任何软件参与
内核中的 DPB helper 模块集中管理

2. Request API 中，一个 request 提交时必须包含什么？

至少一个 OUTPUT buffer 和一个 CAPTURE buffer
至少一个 OUTPUT buffer（可能需要 controls）
仅 controls 即可，buffer 可以独立提交
至少两个 OUTPUT buffer 以形成流水线

3. 多个 slice 属于同一帧时，要使用什么机制？

对每个 slice 单独分配一个 CAPTURE buffer
相同 timestamp + HOLD_CAPTURE_BUF 标志
在用户空间先拼接所有 slice 再提交
使用 V4L2_BUF_FLAG_SLICE 标志分隔

4. Stateful 和 Stateless decoder 在 DRC 时最大的区别是什么？

Stateful 不需要 DRC，stateless 需要
Stateful 由驱动发事件通知，stateless 由用户自行检测并重建
Stateful 要重新分配 buffer，stateless 则不用
两种 decoder 的 DRC 流程完全一样

5. 以下哪个是 stateless decoder 的 coded format 命名特征？

前缀带 RAW_（如 RAW_H264）
前缀带 STATELESS_（如 STATELESS_H264）
后缀带 _SLICE 或 _FRAME（如 H264_SLICE）
格式名字全部大写（如 H264 而非 h264）
