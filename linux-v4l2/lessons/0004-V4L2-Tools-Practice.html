0. 环境准备

如果你有一台 Linux 机器（含内核 5.x+），安装 v4l-utils：

# Debian/Ubuntu
sudo apt install v4l-utils

# Fedora
sudo dnf install v4l-utils

# Arch
sudo pacman -S v4l-utils

如果你在 macOS（像本课程的目标环境），需要借助 QEMU 运行 Linux VM。最简单的方案：

# 用 Ubuntu Cloud Image + QEMU（详见下一节）
# 或使用 multipass:
multipass launch --name v4l2-lab --cpus 2 --mem 2G --disk 10G
multipass shell v4l2-lab
# 在 VM 中:
sudo apt update && sudo apt install v4l-utils linux-modules-extra-$(uname -r)
本课可以只读不练。所有命令和输出都已展示，作为你登录 Linux 时的操作手册。核心目标：让你看懂 v4l2-ctl 的输出、会用 v4l2-compliance 排查问题。

1. v4l2-ctl：瑞士军刀

v4l2-ctl 是 v4l-utils 中的核心工具，覆盖了几乎所有 V4L2 ioctl 的命令行包装。下面按使用场景归类。

1.1 设备发现

列出所有 V4L2 设备

v4l2-ctl --list-devices
输出每个 /dev/videoX 节点的设备名和路径。

查看设备详细信息

v4l2-ctl -d /dev/video0 --all
一口气输出：driver/card/bus info、所有 format、controls、selection rectangles 等。最常用的排查命令。

1.2 格式操作

列出 CAPTURE 支持的格式

v4l2-ctl -d /dev/video0 --list-formats
输出如 [0]: 'H264' (H.264, compressed) — 索引、FourCC、描述。

列出 OUTPUT 支持的格式（decoder 关键！）

v4l2-ctl -d /dev/video0 --list-formats-out
decoder 的 OUTPUT queue 是 coded format 入口，用这个查看支持哪些编码格式。

获取/设置 OUTPUT 格式

# 获取当前 OUTPUT 格式
v4l2-ctl -d /dev/video0 --get-fmt-video-out

# 设置 OUTPUT 为 H.264 1920x1080
v4l2-ctl -d /dev/video0 --set-fmt-video-out \
  width=1920,height=1080,pixelformat=H264
这等同于 S_FMT(OUTPUT)。对应 stateful decoder 初始化第一步。

获取/设置 CAPTURE 格式

v4l2-ctl -d /dev/video0 --get-fmt-video
v4l2-ctl -d /dev/video0 --set-fmt-video \
  width=1920,height=1088,pixelformat=NV12

1.3 Controls 操作（stateless decoder 关键！）

Controls 是 V4L2 中配置参数的标准方式。对 stateless decoder，所有解码参数（SPS/PPS/slice param/decode param）都是 controls。

列出所有 controls

v4l2-ctl -d /dev/video0 --list-ctrls

# 只看特定类型的 controls
v4l2-ctl -d /dev/video0 --list-ctrls-menu
输出 controls 的 ID、类型、范围、当前值。stateless decoder 会有大量 V4L2_CID_STATELESS_* controls。

设置 control 值

# 设置整数 control
v4l2-ctl -d /dev/video0 --set-ctrl brightness=128

# 设置 menu control
v4l2-ctl -d /dev/video0 --set-ctrl \
  video_bitrate_mode=0
简单 controls 可命令行设置。复杂 structure controls（如 H.264 SPS）需要用代码或脚本。

1.4 Buffer 与 Streaming 操作

测试 streaming（MMAP 模式）

# CAPTURE: 抓 10 帧，保存到文件
v4l2-ctl -d /dev/video0 --stream-mmap --stream-count=10 \
  --stream-to=capture.raw

# OUTPUT: 从文件读入数据流送给 OUTPUT queue
v4l2-ctl -d /dev/video0 --out-mmap --stream-count=10 \
  --stream-from=input.h264

# 同时 OUTPUT → CAPTURE (mem2mem)
v4l2-ctl -d /dev/video0 --out-mmap \
  --stream-from=input.h264 \
  --stream-mmap --stream-to=decoded.yuv
相当于完整走一遍 REQBUFS → QBUF → STREAMON → DQBUF → STREAMOFF 流程。

1.5 事件订阅（stateful decoder DRC/drain）

订阅 SOURCE_CHANGE 事件

v4l2-ctl -d /dev/video0 --subscribe-event source_change

# 等待事件（阻塞）
v4l2-ctl -d /dev/video0 --wait-event source_change
对应 stateful decoder 中 V4L2_EVENT_SOURCE_CHANGE 的订阅/等待。DRC 检测不可或缺。

1.6 Decoder 专用命令

发送 STOP/START 命令（stateful drain/seek）

# 触发 drain
v4l2-ctl -d /dev/video0 --decoder-cmd stop

# 恢复解码
v4l2-ctl -d /dev/video0 --decoder-cmd start

# flush（stateless decoder 释放 held buffer）
v4l2-ctl -d /dev/video0 --decoder-cmd flush
发送 VIDIOC_DECODER_CMD，对应 stateful 的 drain/seek 和 stateless 的 flush。

尝试解码一帧（需要真实硬件或 vicodec）

# 用 GStreamer + v4l2sl 插件（stateless）
gst-launch-1.0 filesrc location=test.h264 ! h264parse ! \
  v4l2slh264dec ! videoconvert ! autovideosink

# 或用 stateful decoder
gst-launch-1.0 filesrc location=test.h264 ! h264parse ! \
  v4l2h264dec ! videoconvert ! autovideosink
v4l2slh264dec 是 stateless 插件，v4l2h264dec 是 stateful 插件。

2. v4l2-compliance：驱动一致性测试

v4l2-compliance 是 V4L2 驱动的自动化测试套件。它对驱动执行所有标准 ioctl 并验证返回值、边界条件、状态机正确性。这是排查驱动 bug 的第一站。

基本兼容性测试

v4l2-compliance -d /dev/video0
测试 open/close、querycap、枚举 formats、controls、selection API 等。

含 streaming 的完整测试

v4l2-compliance -d /dev/video0 -s
在 -s 模式下会实际分配 buffer、启动 streaming、读写数据，验证 QBUF/DQBUF/STREAMON/STREAMOFF 的完整流程。

mem2mem 设备测试

v4l2-compliance -d /dev/video0 -m2m
让工具理解这是一对 OUTPUT+CAPTURE 队列，执行 decoder 特有的测试。

针对 vivid 的完整测试（关闭错误注入）

# vivid 需要关闭错误注入，否则 disconnect 会导致后续全部失败
sudo modprobe vivid no_error_inj=1
v4l2-compliance -d /dev/video0 -s

解读输出

典型输出结构：

Compliance test for vivid device /dev/video0:

Driver Info:
        Driver name      : vivid
        Card type        : vivid
        ...

Total for vivid device /dev/video0: 47, Succeeded: 47, Failed: 0, Warnings: 0

遇到 FAIL 时看具体测试名，结合内核文档中的对应 ioctl 章节排查。

3. 虚拟驱动：vivid 与 vicodec

3.1 vivid：通用视频设备模拟器

vivid 模拟 视频采集/输出/radio/VBI 等设备。虽然它不是 decoder，但对练习基本的 V4L2 操作（格式设置、buffer 管理、streaming）非常有价值。

加载 vivid

# 加载 vivid（创建 capture + output 两个 video 节点）
sudo modprobe vivid

# 查看创建了哪些节点
v4l2-ctl --list-devices

# vivid 常见输出:
# vivid (platform:vivid-000):
#         /dev/video0   ← video capture
#         /dev/video1   ← video output
#         /dev/vbi0     ← vbi capture
#         /dev/vbi1     ← vbi output
#         /dev/radio0   ← radio rx
#         /dev/radio1   ← radio tx
#         /dev/swradio0 ← SDR

设置多平面模式

sudo modprobe vivid multiplanar=2
multiplanar=2 让设备使用 MPLANE API，这是 decoder 的常用模式。

启用 Request API（模拟 stateless 设备行为）

sudo modprobe vivid supports_requests=2
supports_requests=2 表示 requires requests，强制使用 Request API。

3.2 vicodec：Stateless 虚拟解码器

vicodec 是内核自带的虚拟 stateless codec 驱动 (FWHT codec)，能模拟完整的 stateless decoder 行为：

加载 vicodec

sudo modprobe vicodec

# 查看设备:
v4l2-ctl --list-devices
# vicodec (platform:vicodec):
#         /dev/video2   ← mem2mem stateless decoder

# 查看它的 coded format:
v4l2-ctl -d /dev/video2 --list-formats-out
# ioctl: VIDIOC_ENUM_FMT
#         Type: Video Output
#         [0]: 'FWHT' (FWHT, compressed)
vicodec 使用 FWHT (Fast Walsh-Hadamard Transform) 作为编码格式——不是真实的 H.264/HEVC，但完整的 stateless API 都在。

用 v4l2-compliance 测试 vicodec

v4l2-compliance -d /dev/video2 -m2m -s

3.3 visl：Stateless 解码器测试框架

从 Linux 5.19+ 开始还有 visl (Virtual Stateless Decoder)，支持更多 codec 格式（H.264/H.265/VP8/VP9 等）：

sudo modprobe visl

# visl 会暴露所有它支持的 coded format
v4l2-ctl -d /dev/video3 --list-formats-out
visl 是纯测试框架，不会真的解码出正确的像素数据——它只是验证 API 流程的正确性。

4. 实战工作流：排查一个 decoder 问题

假设你接到一个 bug：某 stateful decoder 在分辨率从 1080p 切换到 720p 后，解码器卡住不再产出帧。

确认设备
v4l2-ctl --list-devices 找到目标 decoder 的 /dev/videoX。
全面信息采集
v4l2-ctl -d /dev/videoX --all 看 driver info、coded format、controls。
跑合规测试
v4l2-compliance -d /dev/videoX -m2m -s 排除基本 API 使用错误。
检查事件
如果 stateful：用 --subscribe-event source_change 订阅，看驱动是否发出事件。
最小复现
写一个最小用户程序或用 v4l2-ctl --stream-mmap 走一遍 DRC 模拟流程。
看内核日志
dmesg | grep -i v4l2 / dmesg | grep -i vicodec 查驱动打印。

6. 小测验

1. 用 v4l2-ctl 设置 OUTPUT queue 的格式，正确的选项是？

--set-fmt-video width=1920,height=1080
--set-fmt-video-out width=1920,height=1080,pixelformat=H264
--set-fmt-video-capture width=1920,height=1080
--set-fmt-video-m2m width=1920,height=1080

2. v4l2-compliance 的 -s 选项代表什么？

只做静态测试，不触碰 streaming
跳过所有 controls 相关测试
包含 streaming 测试（实际分配 buffer 并读写）
以静默模式运行，不打印输出

3. vivid 和 vicodec 的区别是什么？

vivid 做解码，vicodec 做采集
两个完全相同，只是名字不同
vivid 模拟采集/输出设备，vicodec 模拟 stateless decoder
vivid 是 stateful，vicodec 是 encoder

4. 要触发 stateful decoder 的 drain，v4l2-ctl 用什么命令？

--stream-mmap --stream-count=0
--set-ctrl drain=1
--decoder-cmd stop
--subscribe-event eos

5. 排查 DRC 卡死问题时，优先检查什么？

重新编译内核
订阅 source_change 事件，确认驱动是否发出
用 v4l2-compliance 测试其他无关设备
直接重启设备
