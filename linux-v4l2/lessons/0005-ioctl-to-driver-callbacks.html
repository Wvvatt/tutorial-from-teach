Lesson 5

从 ioctl 追到驱动回调

把用户态的 S_FMT/QBUF/STREAMON/DQBUF 映射到 v4l2_m2m、videobuf2 和驱动核心函数
← Lesson 4: V4L2 调试工具实战  |  参考：m2m/vb2 回调速查

0. 本课目标

前几课你已经会从用户态理解 decoder：OUTPUT 放 coded bitstream，CAPTURE 取 decoded frame。现在要跨过源码阅读的第一道坎：看到一个驱动时，知道从哪里开始追。

今天的单一胜利：拿到一个 V4L2 decoder 驱动后，你能先找 3 组表：v4l2_ioctl_ops、vb2_ops、v4l2_m2m_ops，再把用户态 ioctl 对应到具体驱动回调。

1. 四层 mental model

userspace ioctl→ v4l2_ioctl_ops→ v4l2_m2m helper→ vb2_ops / m2m_ops→ hardware job

这不是精确 call graph，而是阅读源码时最有用的地图。大部分 mem2mem decoder 都把通用 buffer 逻辑交给 V4L2 mem2mem framework 和 videobuf2，驱动只实现硬件相关的格式、queue、启动、完成逻辑。

2. ioctl 到驱动：先看这张表

用户态动作常见 ioctl_ops 入口最终关注点
设置 coded/raw formatvidioc_s_fmt_vid_out_mplane
vidioc_s_fmt_vid_cap_mplane		驱动如何保存 OUTPUT/CAPTURE format；OUTPUT 是否会重置 CAPTURE
分配 bufferv4l2_m2m_ioctl_reqbufs 或 wrappervb2_ops.queue_setup 计算 plane 数量和大小
QBUFv4l2_m2m_ioctl_qbufvb2_ops.buf_prepare 验证 buffer；buf_queue 放入 ready queue
STREAMONv4l2_m2m_ioctl_streamonvb2_ops.start_streaming 启动队列；m2m 尝试调度 job
硬件开始处理不是直接 ioctlv4l2_m2m_ops.device_run 写寄存器 / 提交 firmware 命令
DQBUFv4l2_m2m_ioctl_dqbuf中断或 workqueue 调用 vb2_buffer_done 后，用户才能 DQBUF
STREAMOFFv4l2_m2m_ioctl_streamoffvb2_ops.stop_streaming 停硬件并归还所有 buffer
不要从 DQBUF 开始读驱动。真正的完成点通常在中断 handler、firmware callback 或 workqueue 里，那里会调用 vb2_buffer_done()、v4l2_m2m_buf_done_and_job_finish() 或 v4l2_m2m_job_finish()。

3. 一次 decode job 的典型路径

userspace:
  QBUF(OUTPUT bitstream)
  QBUF(CAPTURE empty frame buffer)
  STREAMON(OUTPUT)
  STREAMON(CAPTURE)

driver / framework:
  ioctl_ops.qbuf
    -> v4l2_m2m_qbuf()
      -> vb2_core_qbuf()
        -> vb2_ops.buf_prepare()
        -> vb2_ops.buf_queue()
          -> v4l2_m2m_buf_queue()
          -> v4l2_m2m_try_schedule()

  if OUTPUT ready && CAPTURE ready && streaming on:
    -> v4l2_m2m_ops.device_run(ctx)
       driver programs hardware

  interrupt / completion:
    driver marks src/dst buffers DONE
    -> v4l2_m2m_buf_done_and_job_finish(...)
       or vb2_buffer_done(...) + v4l2_m2m_job_finish(...)

userspace:
  DQBUF(OUTPUT)   // input buffer returned
  DQBUF(CAPTURE)  // decoded frame returned

v4l2_m2m_try_schedule() 的基础条件来自内核文档：至少一个 source buffer、至少一个 destination buffer、streaming 已开启。driver 还可以通过 job_ready 加额外条件。

4. 读一个驱动时的 6 个搜索词

搜索词你要确认什么
struct v4l2_ioctl_ops用户态 ioctl 入口。格式设置、buffer 操作、decoder_cmd 是否直接用 m2m helper。
struct vb2_opsbuffer 生命周期。重点看 queue_setup、buf_prepare、start_streaming、stop_streaming、buf_queue。
struct v4l2_m2m_ops调度入口。重点看 device_run 和 job_ready。
v4l2_m2m_ctx_initopen() 时如何创建 per-filehandle decode context，以及 queue_init 在哪里。
v4l2_m2m_buf_done硬件完成后在哪里归还 OUTPUT/CAPTURE buffer。
V4L2_BUF_FLAG_LASTdrain/DRC/seek 时最后一个 CAPTURE buffer 如何标记。

5. 小练习：看到代码先归类

想象你在驱动里看到这段代码：

static const struct v4l2_m2m_ops foo_m2m_ops = {
    .device_run = foo_device_run,
    .job_ready  = foo_job_ready,
    .job_abort  = foo_job_abort,
};

你的第一反应应该是：foo_device_run() 是一次硬件 transaction 的入口。它通常会取出当前 OUTPUT/CAPTURE buffer，写寄存器或提交 firmware，然后立即返回；真正完成在中断或异步回调里。
