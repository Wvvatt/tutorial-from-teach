0. 本课目标

今天不读 FWHT 算法，只回答一个工程问题：拿到 decoder 驱动后，怎样在十分钟内找到设备注册、队列配置、job 调度和 buffer 完成四个位置。

单一胜利：用五个符号串起主路径：vicodec_probe → queue_init → buf_queue → job_ready/device_run → v4l2_m2m_job_finish。

1. 先看驱动注册了什么

当前内核的 vicodec-core.c 是测试 codec framework 的虚拟设备。它没有真实硬件寄存器，而是用内存模拟编码/解码；但保留了真实 mem2mem 驱动需要的结构。

vicodec_probe()
  ├─ register_instance("stateful-encoder",  true,  false)
  ├─ register_instance("stateful-decoder",  false, false)
  └─ register_instance("stateless-decoder", false, true)

register_instance()
  ├─ v4l2_m2m_init(&m2m_ops)
  ├─ vdev->ioctl_ops = &vicodec_ioctl_ops
  └─ video_register_device(...)

同一个驱动可以暴露多个 video node；stateful/stateless 的差异会落到每个 instance 的 ctx->is_stateless 分支。

不要混淆：vicodec_dev 是设备级对象；vicodec_ctx 保存一个 file handle 的 queue format、codec state 和 m2m context。

2. queue_init：两条 vb2 队列的出生证明

open 路径通过 v4l2_m2m_ctx_init(..., queue_init) 创建 per-filehandle 的 OUTPUT/CAPTURE 队列。两条队列共用 vicodec_qops，但 type 不同。

字段vicodec 设置阅读意义
typeOUTPUT / CAPTURE（可选 MPLANE）输入码流 / 解码帧
ops&vicodec_qopsqueue_setup、buf_prepare、buf_queue 等回调
mem_opsvb2_vmalloc_memops虚拟驱动用 vmalloc；真实硬件常用 DMA allocator
supports_requestsstateless 为 true允许 controls 与 buffer 绑定到 Request
requires_requestsstateless 为 true每个 stateless 输入 buffer 都必须有 Request

这就是上一课的源码落点：v4l2_m2m 组合两条 vb2 queue，驱动通过 queue_init 注入自己的 queue policy。

3. buf_queue → job_ready：QBUF 不等于马上运行

QBUF 会走到 vicodec_buf_queue()，通常再调用 v4l2_m2m_buf_queue()。这只表示 buffer ready；m2m framework 还要判断 job 能否运行。

QBUF→buf_prepare→buf_queue→try_schedule→job_ready

vicodec 的 job_ready() 展示了 decoder 的现实：即使两条 queue 都有 buffer，也可能还不能跑。stateful decoder 需要先扫描完整 frame header；等待更多码流时返回 0。

static const struct v4l2_m2m_ops m2m_ops = {
    .device_run = device_run,
    .job_ready  = job_ready,
};

job_ready(ctx):
  if (source_changed) return 0;
  if (stateless || encoder || complete_frame_ready) return 1;
  scan OUTPUT for a complete frame;
  return frame_ready ? 1 : 0;

调试“QBUF 成功但没有 DQBUF”时，除了检查 streaming 和 buffer 数量，还要看 job_ready 是否因 header、DRC 或 Request 条件拒绝调度。

4. device_run：一次 transaction 的核心

device_run() 是 m2m framework 把设备所有权交给当前 context 后调用的入口。vicodec 在这里取 source/destination buffer，执行 device_process()，设置 payload、sequence、metadata 和 flags。

device_run(ctx)
  ├─ src = v4l2_m2m_next_src_buf(ctx->m2m_ctx)
  ├─ dst = v4l2_m2m_dst_buf_remove(ctx->m2m_ctx)
  ├─ device_process(ctx, src, dst)
  ├─ v4l2_m2m_buf_copy_metadata(src, dst)
  ├─ v4l2_m2m_buf_done(src, state)
  ├─ v4l2_m2m_buf_done(dst, state)
  └─ v4l2_m2m_job_finish(m2m_dev, ctx->m2m_ctx)

真实硬件驱动通常把第三步替换为“写寄存器/提交 firmware”，然后在中断或异步 callback 中完成最后三步。DQBUF 只是消费已经被 buffer_done 标记的 buffer。

5. 把 stateless Request API 放回源码

stateless instance 在 queue_init 中设置 requires_requests = true。在 device_process 中，它从 source buffer 找到 req_obj.req，调用 v4l2_ctrl_request_setup()，把本次 frame 的 codec controls 装载到 context。

stateless OUTPUT buffer
  └─ req_obj.req
       └─ v4l2_ctrl_request_setup(req, &ctx->hdl)
            └─ 本帧解码参数

完成后：v4l2_ctrl_request_complete(req, &ctx->hdl)

这对应第三课的模型：Request 是“这个 coded buffer 与这组 controls”的原子绑定。驱动从当前 buffer 关联的 Request 取参数，而不是从跨帧全局变量猜参数。

6. DRC 与 drain：在完成路径找协议语义

现象源码信号用户态含义
分辨率变化source_changed，暂缓 job_ready处理 source-change event，重新配置 CAPTURE
最后一帧CAPTURE 加 V4L2_BUF_FLAG_LAST，排队 EOS event用户识别 drain 结束
Request 完成v4l2_ctrl_request_complete()本帧 controls 生命周期结束

读真实 driver 时，先搜索 V4L2_BUF_FLAG_LAST、V4L2_EVENT_SOURCE_CHANGE 和 v4l2_ctrl_request_complete，通常比从格式协商开始更快找到 decoder 的协议语义。

7. 小测验

1. stateless 队列为什么设置 requires_requests？

为了强制使用 MMAP
要求每个输入 buffer 绑定 Request
为了禁止 CAPTURE queue
为了自动开启 drain

2. QBUF 成功但没有立即执行 job，最值得先检查什么？

video_device 的名字
job_ready 的附加条件
CAPTURE 的 timestamp 格式
驱动模块作者

3. DQBUF 能拿到 buffer，驱动侧通常先做了什么？

重新调用 S_FMT
销毁 v4l2_m2m_ctx
调用 buffer_done 标记完成
卸载 vicodec 模块

4. 哪个对象代表一次 open 的独立解码上下文？

vicodec_ctx
pixfmt_info
platform_device
